# AGENTS.md

Короткий контракт работы агента с репозиторием «Бухта».

## 1. Язык и стиль работы

- Объяснения в чате давать на русском языке.
- Перед изменениями читать минимально нужные документы, а не весь каталог.
- Не добавлять абстракции, фреймворки и процессы без понятной пользы для текущей CRM.
- Для новых библиотек выбирать последние стабильные версии, если нет явной причины закрепить более старую.
- Если документация противоречит коду и тестам, считать код и тесты источником истины, затем исправлять документацию.

## 2. Порядок источников истины

1. Код и тесты.
2. Схема базы данных, runtime-контракты и API handlers.
3. `docs/crm-requirements.md` — бизнес-требования и доменная логика.
4. `docs/ARCHITECTURE-PRINCIPLES.md` — стабильные инженерные принципы.
5. `docs/TECH-STACK.md` — выбранный стек и dev-contour.
6. Остальная документация из `docs/`.

Пока код приложения не создан, главным источником истины являются `docs/crm-requirements.md` и решения, явно зафиксированные в документации.

## 3. Документы, которые читать в первую очередь

- Для бизнес-логики CRM: `docs/crm-requirements.md`.
- Для общих архитектурных правил: `docs/ARCHITECTURE-PRINCIPLES.md`.
- Для выбора технологий и Docker-dev подхода: `docs/TECH-STACK.md`.
- Для карты документации: `docs/DOCS-INDEX.md`.
- Для правил планирования задач: `docs/PLANS.md`.
- Для диаграммы процесса: `docs/diagrams/business-flow.d2` или `docs/diagrams/business-flow.png`.

## 4. Инженерные ожидания

- Выбирать простые решения, которые легко читать, тестировать и менять.
- Не строить SaaS, multi-tenant платформу, сложные очереди или микросервисы без явной необходимости.
- Доменные инварианты защищать в коде и тестах, а не только в UI.
- Денежные и товарные операции делать транзакционно.
- Историю действий проектировать как проверяемый журнал: исправления новой операцией, без подмены прошлого.
- Роли и права описывать явно; не зашивать бизнес-решения в случайные проверки UI.
- Зависимости держать на актуальных стабильных версиях и учитывать известные уязвимости перед добавлением или обновлением пакетов.

## 5. Документационная гигиена

- Документация ведется на русском языке.
- Новый документ добавлять в `docs/DOCS-INDEX.md`.
- Любое изменение поведения, доменной модели, прав, API, тестового контура или архитектурного решения должно сопровождаться обновлением соответствующей документации.
- Если код меняется без изменения документации, в финальном ответе нужно явно сказать, почему документация не требовалась.
- Устаревшие утверждения удалять, если они не являются активным решением или явно отложенным вопросом.
- Не дублировать один и тот же смысл в нескольких документах без причины.
- `docs/crm-requirements.md` хранит бизнес-требования, а не технический backlog.
- `docs/ARCHITECTURE-PRINCIPLES.md` хранит стабильные принципы, а не историю обсуждений.
- `docs/DECISIONS.md` хранит короткие принятые архитектурные решения.
- `docs/PLANS.md` хранит правила работы с планами, а не сами активные планы.
- После появления docs harness команда проверки документации должна входить в обязательную верификацию перед коммитом и PR.
- Целевой docs harness должен проверять минимум:
  - ссылки на документы и файлы;
  - наличие документов в `docs/DOCS-INDEX.md`;
  - запрет терминов и артефактов старых проектов;
  - обязательные разделы в ключевых SoR-документах.

## 6. Планы и lifecycle задач

- Для сложной инициативы использовать execution plan по правилам `docs/PLANS.md`.
- Active plans хранить в `docs/exec-plans/active/`.
- После завершения переносить план в `docs/exec-plans/completed/` и фиксировать выполненные проверки.
- Если план заменен другим решением, переносить его в completed со статусом `Superseded`.
- Отложенные идеи, которые не являются багами или активной работой, писать в `docs/exec-plans/deferred-roadmap.md`.
- Признанный техдолг писать в `docs/exec-plans/tech-debt-tracker.md`.
- Не хранить бизнес-требования только в планах: постоянные правила переносить в `docs/crm-requirements.md`, архитектурные решения — в `docs/DECISIONS.md` или профильный документ.
- Если задача маленькая, отдельный plan-файл не нужен; достаточно обновить затронутые SoR-документы и дать финальное summary.

## 7. Тестирование

- Мы стремимся покрывать тестами весь значимый код, особенно доменную логику, деньги, остатки, права и историю действий.
- Новый доменный код пишется вместе с тестами. Рефакторинг критичного кода начинается с targeted safety-net тестов текущего поведения.
- Минимальный набор для доменной операции: happy path, запрет отрицательного остатка, проверка прав, запись истории.
- Для критичных денежных сценариев тестировать наличный и безналичный варианты отдельно.
- Для сложной работы использовать цикл: targeted tests -> изменение -> targeted verification -> полный релевантный прогон.
- Перед завершением задачи запускать максимально полный доступный набор проверок: lint, typecheck, unit/integration tests, smoke и docs check. Если часть проверок невозможна, указать причину.
- Если тестовый стек еще не выбран, сначала описать ожидаемые сценарии тестирования в плане реализации.
