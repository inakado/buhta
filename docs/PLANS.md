# PLANS

Правила для execution plans и рабочих инициатив проекта «Бухта».

## 1. Когда нужен execution plan

План нужен, если задача затрагивает хотя бы одно:

- несколько доменных операций или ролей;
- миграции БД;
- изменение прав доступа;
- изменение денежных или товарных инвариантов;
- существенный API/contract change;
- крупный refactor;
- новый dev/test/docs harness.

Для маленькой точечной правки достаточно описания в задаче и финального summary.

## 2. Формат плана

План должен содержать:

- цель;
- scope;
- out of scope;
- затронутые документы;
- затронутые модули;
- шаги реализации;
- тестовый план;
- риски и rollback;
- открытые вопросы.

## 3. Где хранить планы

Целевая структура после очистки:

- `docs/exec-plans/active/` — активные планы;
- `docs/exec-plans/completed/` — завершенные планы;
- `docs/exec-plans/deferred-roadmap.md` — отложенные идеи;
- `docs/exec-plans/tech-debt-tracker.md` — признанный техдолг.

План можно вести в чате только для небольших задач. Для сложной инициативы заводить файл в `docs/exec-plans/active/`, а после завершения переносить его в `docs/exec-plans/completed/`.

Для v1 используется один долгоживущий meta-plan/roadmap:

- `docs/exec-plans/active/2026-05-27-v1-roadmap.md`

Этот документ остается активным до завершения v1 и описывает последовательность этапов на верхнем уровне. Он не заменяет детальные execution plans. Для каждого крупного этапа v1 создается отдельный plan-файл в `docs/exec-plans/active/`, который после завершения переносится в `docs/exec-plans/completed/`.

## 4. Lifecycle

Статусы:

- `Draft` — план обсуждается.
- `Active` — по плану идет реализация.
- `Active / Meta-plan` — долгоживущая roadmap, которая остается активной на протяжении крупной инициативы и координирует отдельные execution plans.
- `Completed` — задача завершена и проверена.
- `Superseded` — план заменен другим решением.
- `Deferred` — отложено осознанно.

## 5. Правила обновления

- Если во время реализации меняется доменное решение, сначала обновить `docs/crm-requirements.md` или явно зафиксировать решение в плане.
- Если меняется стек, структура или проверочный контур, обновить `docs/TECH-STACK.md`, `docs/ARCHITECTURE-PRINCIPLES.md` или `docs/DEVELOPMENT.md`.
- После завершения крупной задачи в плане должен быть список фактически выполненных проверок.
- Устаревшие планы не должны подменять актуальные SoR-документы.

## 6. Что не хранить в планах

- постоянные бизнес-требования;
- текущую схему ролей;
- runbook команд;
- секреты и env-значения;
- длинные логи команд.

Для этого есть отдельные документы и код.
