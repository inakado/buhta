# ARCHITECTURE-PRINCIPLES

Стабильные инженерные принципы проекта «Бухта».

Выбранное направление стека описано в `docs/TECH-STACK.md`: `Next.js` для web, `NestJS` для backend, `PostgreSQL`, `Prisma`, `zod`, `pnpm workspace`.

Ожидаемое направление auth: `BetterAuth` как предпочтительный кандидат для стандартной аутентификации, с обязательной проверкой интеграции в связке `NestJS + Prisma + роли Бухты` перед финальным scaffold приложения.

## 1. Главный вектор

Проект — внутренняя CRM для небольшой команды. Нам нужна простая, надежная система учета продукции, денег, продаж и действий пользователей.

Первая версия разрабатывается как цельный продукт до пользовательского тестирования. Мы не планируем частичные production-релизы отдельных slices пользователям, но каждый крупный этап обязан закрываться внутренним инженерным checkpoint: автотесты, релевантные проверки, обновленная документация и ручная проверка UI владельцем проекта.

Мы выбираем решения, которые:

- легко читать и объяснять;
- легко тестировать;
- можно менять при изменении бизнес-логики;
- не требуют лишней инфраструктуры;
- не превращают внутренний инструмент в SaaS-платформу раньше времени.

## 2. Простота без хрупкости

- Начинать с модульного монолита, если нет сильной причины делать иначе.
- Разделять доменную логику, доступ к данным и UI, но не плодить слои ради слоев.
- Предпочитать явный код скрытой магии.
- Не добавлять очередь, event bus, CQRS или отдельный сервис, если задачу закрывает обычная транзакция.
- Абстракцию добавлять только после появления реального повторения или сложного правила, а не заранее.
- Использовать зрелые boring dependencies и не тратить “инновационные токены” на инфраструктуру, которая не нужна внутренней CRM.
- Использовать последние стабильные версии библиотек и фреймворков, если нет зафиксированной причины закрепить более старую версию.
- Не оставлять известные high/critical уязвимости в зависимостях без явного решения, mitigation или временного исключения с причиной.
- Держать complexity budget: файл, модуль и handler должны иметь одну главную ответственность и оставаться читаемыми.

## 3. Архитектурные правила кода

- **SRP + complexity budget**: модуль и файл имеют одну главную ответственность; большие orchestration-блоки декомпозируются до понятных units.
- **Contract-first**: transport/runtime-контракты описываются один раз и переиспользуются между API и web.
- **Fail-fast boundary validation**: все внешние входы валидируются на boundary до попадания в доменные сервисы.
- **Read/write separation без CQRS-фанатизма**: read-path и write-path не смешиваются в одном orchestration без явной причины.
- **Typed mapping**: преобразования DB/API/UI выполняются явными mapper-функциями без протекания `any` и неконтролируемого `unknown`.
- **Unified error semantics**: пользовательские и интеграционные ошибки проходят через единый `code/message/details` contract.
- **Policy-as-code**: права, лимиты операций, правила дисконта, запрет минусов и допустимые статусы оформляются как явные политики/helpers, а не размазываются по UI и handlers.
- **Convention over duplication**: повторяющиеся cross-cutting patterns оформляются как общие примитивы, если повторение уже стало реальным.
- **Dependency rule enforcement**: после выбора структуры границы модулей должны проверяться автоматически, а не только договоренностью.
- **Typed operation history**: общий журнал операции фиксирует факт действия, а детали важных операций описываются типизированно, чтобы отчеты, тесты и инварианты не зависели от произвольного JSON.

## 4. Frontend rules

- **Mobile-first product surface**: ежедневные операционные сценарии проектируются сначала для телефона; desktop/tablet оптимизируются для аудита, статистики и широких отчетов.
- **PWA/app-like UX**: web-приложение должно ощущаться как установленное приложение, но без усложнения доменной логики ради offline-first.
- **Fast daily operations**: частые действия должны занимать 2-3 основных касания, если бизнес-правило не требует дополнительного подтверждения.
- **Online-only writes by default**: операции, меняющие деньги, остатки, статусы или историю, выполняются только online, пока отдельно не спроектирован безопасный offline-sync.
- **Server-state discipline**: серверные данные живут в выделенном server-state слое, а не в произвольных `useState/useEffect`.
- **Effect isolation**: async orchestration и побочные эффекты выносятся в hooks/helper-слои; UI остается декларативным.
- **No hidden business logic in frontend**: frontend может считать preview и помогать пользователю, но финальные доменные решения принимает backend/domain layer.
- **Server-first rendering by default**: если выбираем Next.js, client boundaries вводятся только там, где нужна интерактивность.
- Формы не должны напрямую реализовывать правила движения денег, товара, дисконтов и корректировок.

## 5. Доменная модель важнее CRUD

Критичные правила должны жить в доменной/application-логике и проверяться тестами:

- цена партии фиксируется при выпуске;
- дисконт назначается заранее на конкретное количество остатка и не меняет цену партии;
- нельзя продать, загрузить или сгрузить больше доступного остатка;
- наличная продажа увеличивает наличный баланс источника;
- безналичная продажа не увеличивает наличный баланс;
- курьерские наличные попадают на распределитель только при сгрузке;
- проведенные операции не редактируются задним числом;
- исправления оформляются отдельной операцией.

UI может помогать пользователю, но не является источником защиты инвариантов.

## 6. Транзакции, конкуренция и история

- Любая операция, меняющая товарный или денежный баланс, должна выполняться атомарно.
- Движение денег/товара оформляется как command handler: проверить права и входные данные, выполнить доменные проверки, изменить баланс, записать операцию и журнал в одной транзакции.
- Операции продажи, загрузки курьера, сгрузки, списания денег и корректировки должны быть защищены от двойного выполнения из-за повторного запроса или двойного клика.
- Одновременные операции не должны позволять продать, загрузить или сгрузить один и тот же остаток дважды.
- Операция должна записывать факт действия в журнал.
- Журнал операций проектировать как общий operation/audit envelope: кто, когда, какой тип действия выполнил.
- Для важных доменных операций использовать типизированные details/models: продажа, загрузка курьера, сгрузка, выпуск продукции, списание, дисконт, корректировка.
- Не складывать все детали критичных операций в один универсальный JSON payload, если эти данные нужны для инвариантов, отчетов или тестов.
- Журнал действий не должен редактироваться или удаляться обычными бизнес-операциями.
- История продаж, движений товара, денежных операций и корректировок проектируется append-only.
- Для корректировок хранить ссылку на исходную операцию и причину.
- Баланс можно кэшировать или хранить как проекцию, но источник правды должен быть восстановим из операций либо защищен инвариантами и тестами.
- Отчеты строятся из фактов операций или надежных проекций, а не из UI-состояния.
- Для критичных сущностей использовать archive/soft delete вместо физического удаления, если удаление может сломать историю.

## 7. Права доступа

- Права задавать явно через роли и разрешения.
- Права реализовывать через единый backend policy layer, чтобы изменение разрешений роли было локальным изменением, а не поиском проверок по controllers и UI.
- Не зашивать бизнес-решения только в UI.
- Если роль может видеть данные, но не менять их, это должно быть отдельным правом.
- Спорные права фиксировать как открытое решение в документации до реализации.
- Auth-библиотека может управлять пользователями, сессиями и базовыми ролями, но доменные разрешения на выпуск продукции, продажи, скидки, списания, загрузку и сгрузку проверяются в backend policy/guards и application services.
- Если используется `BetterAuth`, его access-control возможности можно применять как слой идентификации и грубых permissions, но не как единственный источник доменных инвариантов.

## 8. Контракты, валидация и статусы

- Внешний ввод валидировать на границе системы.
- Ошибки доменных правил возвращать явно: что нарушено и что пользователь может сделать.
- API и UI должны использовать одинаковые понятия домена: партия, распределитель, курьер, продажа, сгрузка, дисконт.
- Не допускать бесконтрольных `any`, неявных преобразований денег и количества.
- Статусы партий, уведомлений, продаж, корректировок и других workflow-сущностей должны иметь явные разрешенные переходы.
- Запрещено произвольно обновлять статус, если такой переход не разрешен доменным правилом.

## 9. Деньги и количество

- Деньги хранить в минимальных единицах валюты или другом явно выбранном числовом формате без ошибок округления.
- Количество хранить с учетом единицы измерения.
- Все вычисления суммы продажи, дисконта и балансов должны быть детерминированными.
- Денежные отчеты должны отличать валовую сумму, сумму дисконта, фактическую выручку, наличные и безналичные продажи.

## 10. Тестирование

Цель проекта — высокий уровень покрытия значимого поведения тестами. Особенно важны не проценты покрытия сами по себе, а проверка бизнес-инвариантов, денежных сценариев, прав и истории действий.

Минимальный тестовый контур для каждой доменной операции:

- успешный сценарий;
- отказ при недостаточном остатке или недостаточных наличных;
- отказ при недостаточных правах;
- запись истории действий;
- корректное изменение всех затронутых балансов.

Для денежных сценариев отдельно проверять:

- наличную продажу с распределителя;
- безналичную продажу с распределителя;
- наличную продажу курьером;
- сгрузку курьера на распределитель;
- продажу заранее дисконтированного остатка.

Для рефакторинга и изменений в существующем коде:

- сначала добавить или найти targeted tests, которые фиксируют текущее ожидаемое поведение;
- после изменения прогнать targeted verification;
- перед завершением задачи прогнать полный релевантный набор проверок;
- не делать blanket-refactor критичного кода без safety-net.

Целевые уровни тестов после выбора стека:

- unit-тесты для чистых доменных правил и расчетов;
- service/application tests для транзакционных операций;
- integration tests для API boundary, прав доступа и записи истории;
- component/UI tests для основных пользовательских сценариев;
- smoke tests для ключевых end-to-end потоков.

Для frontend дополнительно проверять:

- основные mobile flows для ролей;
- отсутствие горизонтального overflow на мобильных экранах;
- readable loading/error states;
- online-only поведение write-операций при отсутствии соединения;
- PWA manifest/installability после реализации PWA-контура.

## 11. Документация и планы

- `docs/crm-requirements.md` описывает бизнес-логику и открытые вопросы.
- Этот документ описывает стабильные инженерные принципы.
- Архитектурные решения по стеку, модульной структуре и тестам фиксируются отдельно после выбора стека.
- Не хранить в принципах историю обсуждений, прогресс задач и временный backlog.

Документация должна обновляться вместе с изменениями:

- изменение бизнес-логики или прав обновляет `docs/crm-requirements.md`;
- изменение архитектурного решения обновляет архитектурные документы;
- изменение команд разработки, тестов или окружения обновляет development/runbook документ после его появления;
- новый документ добавляется в `docs/DOCS-INDEX.md`;
- если документация не меняется при изменении кода, причина должна быть явной.

После адаптации harness в проекте должна появиться команда проверки документации, аналогичная `docs:check`: проверка ссылок, индекса, обязательных разделов и запретных терминов старых проектов.

## 12. Auth direction

Для проекта предпочтительнее использовать зрелую auth-библиотеку вместо собственной реализации стандартных auth-механик.

`BetterAuth` подходит как основной кандидат, потому что закрывает типовые задачи:

- email/password sign-in;
- сессии и cookies;
- Prisma-интеграцию;
- user management;
- расширяемые роли и permissions;
- клиентскую интеграцию с React/Next.js.

Ожидаемый принцип внедрения:

- `BetterAuth` отвечает за identity, session и базовое управление пользователями.
- Backend отвечает за доменные права и бизнес-инварианты.
- Роли «Бухты» должны быть явно сопоставлены с permissions, но проверки критичных операций должны оставаться на стороне backend.
- Нельзя принимать решение о доступе к денежной или товарной операции только на основании frontend-состояния.

Перед финальным выбором нужно сделать короткий технический spike:

- поднять минимальную связку `NestJS + Prisma + BetterAuth`;
- проверить protected route/controller;
- проверить получение текущей сессии в Nest guard;
- проверить роль пользователя и один permission из матрицы «Бухты»;
- покрыть это integration test;
- убедиться, что отключение body parser или другие требования интеграции не ломают обычные JSON endpoints NestJS.

Если spike показывает чистую интеграцию, `BetterAuth` становится auth-решением по умолчанию. Если интеграция требует обходных решений, auth остается отдельным решением NestJS, но с тем же принципом: не писать самописную auth-инфраструктуру без необходимости.
