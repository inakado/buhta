# Role Home Working Cards

Статус: `Draft`.

## Цель

Переработать только карточки на первых экранах ролей так, чтобы они показывали больше полезных данных в той же общей структуре экрана.

Карточки разных ролей не приводятся к одному шаблону. Общими остаются только ритм, типографика, отступы, состояния загрузки/ошибки и общий mobile shell.

## Scope

- `commercial_manager`: улучшить существующие карточки, не превращая home в новый большой экран.
- `production_manager`: отполировать главный экран, read-only экран `На распределителе`, настройки и placeholder уведомлений без расширения функциональности.
- `distributor_worker`: короткий рабочий экран точки продажи.
- `courier`: личное состояние курьера и два реальных действия.
- `director`: контроль товара и денег по контурам без операционных write-действий.
- CSS-правила для новых карточек в существующей дизайн-системе.
- Компонентные проверки, browser-smoke и документация.

## Out Of Scope

- Новые backend read models.
- Fake revenue, fake latest sales, fake alerts.
- Универсальный `RoleSummaryCard` или builder для всех ролей.
- Новая дизайн-система поверх текущей.
- Desktop/tablet redesign.
- Копирование Dribbble-экранов один-в-один.
- Плитки, которые не ведут к действию.
- Рекомпоновка всего home-экрана в много секций.
- Добавление курьерских списков на commercial home только ради “больше данных”.

## Текущая проблема

- `CommercialManagerHome` и `DistributorWorkerHome` используют общий `DistributorHomeOverview`.
- Из-за этого коммерческий руководитель видит почти тот же экран, что работник распределителя, хотя у него больше прав и шире зона контроля.
- `summary-card` сейчас часто является крупной цветной поверхностью с одним числом и поясняющей строкой.
- В части экранов карточки не показывают, где именно товар/деньги и что требует внимания.
- Первая попытка вынести commercial home в несколько новых секций дала слишком тяжелый экран. Дальше работаем уже: меняем содержание карточек, а не строим новый dashboard.

## Правила Нового Прохода

- Сначала определить ролевое решение, потом верстать карточку.
- Не добавлять карточку, если она не отвечает на вопрос пользователя.
- Не выносить ролевую структуру в `ui/*`.
- `ui/*` может содержать только нейтральные элементы после второго реального использования.
- Названия блоков должны быть предметными: `Распределитель`, `Курьеры`, `Деньги`, `Мой товар`, `В обороте`.
- Не использовать пустые заголовки вроде `Аналитика`, `Сводка`, `Статистика`, если внутри нет конкретного backend-среза.
- Цвет не должен быть главным носителем смысла. Смысл дают число, подпись, разрез и статус.
- Если данных нет в текущем API, блок либо не добавляется, либо явно попадает в отложенный backend этап.
- На один checkpoint менять только 1-2 карточки роли. Списки и навигацию не трогать без отдельной причины.
- Новая карточка не должна увеличивать первый экран больше чем на один компактный блок.

## Целевая Компоновка

### Commercial Manager

Коммерческий home остается в текущей структуре: заголовок, 1-2 карточки, `Продать`, `Остатки`.

Использовать доступные данные:

- `getDistributorInventory`;
- `getDistributorCashBalances`;

Первый экран:

1. `Остаток распределителя`
   - единицы товара;
   - стоимость товара;
   - количество позиций;
   - статус `Можно продавать` / `Нет товара`.

2. `Наличные`
   - наличные на распределителе;
   - статус `Есть операции` / `Без операций`, если это можно определить из `updatedAt`.

3. `Продать`
   - прежняя action tile;
   - не менять стиль на черную кнопку.

4. `Остатки`
   - подробный список не показывается на home;
   - открывается отдельным нижним пунктом `Остатки`.

Не добавлять:

- плитки `Остатки`, `Клиенты`, `Курьеры`;
- отдельные блоки `Курьеры`, `Клиенты`, `Деньги` сверх текущей структуры;
- fake `Продажи сегодня`;
- fake `Последние продажи`;
- декоративные графики без данных.

### Distributor Worker

Экран работника распределителя должен оставаться короче коммерческого.

Первый экран:

1. `Распределитель`
   - единицы товара;
   - стоимость товара;
   - количество позиций;
   - статус “нет товара” при нулевом остатке.

2. `Деньги`
   - наличные на распределителе, если роль имеет permission;
   - статус загрузки/ошибки рядом с числом.

3. `Продать`
   - одна action tile;
   - disabled только при offline или отсутствии права.

4. `Остатки`
   - список товаров остается на home;
   - строки должны показывать товар, владелец/точка, шт, стоимость, цена за единицу.

### Courier

Курьерский home должен отвечать на вопрос “что у меня на руках и что я могу сделать сейчас”.

Первый экран:

1. `Мой товар`
   - единицы;
   - стоимость;
   - количество позиций;
   - статус “пусто”, если товара нет.

2. `Мои деньги`
   - наличные курьера;
   - cashless не добавлять, пока нет отдельного read model.

3. `Продать` и `Загрузить`
   - только реальные action tiles;
   - `Сгрузить` не возвращать до backend flow.

4. `Мой остаток`
   - список товаров;
   - строки должны помогать быстро выбрать товар для продажи.

### Director

Директорский home не должен выглядеть как экран оператора.

Первый экран:

1. `В обороте`
   - всего товара;
   - у распределителя;
   - у курьеров.

2. `Деньги`
   - всего наличных;
   - распределитель;
   - курьеры.

3. `Распределитель`
   - позиции и стоимость;
   - список товаров ниже.

4. `Курьеры`
   - строки курьеров: имя, товар, деньги;
   - ниже общий список товаров у курьеров.

5. Управленческие действия
   - оставить disabled только если они нужны для ориентира будущего backend этапа;
   - не размещать выше реальных данных.

Не добавлять:

- продажу;
- fake revenue;
- fake recent operations;
- графики без данных.

## Визуальные Решения

- Большая lime/green поверхность используется редко: только для главного статуса роли, если внутри больше одного полезного значения.
- Lime/green summary не должен иметь вложенные белые карточки: внутренние значения разделяются расстоянием, типографикой и очень мягкими зелеными линиями.
- Большинство карточек должны быть белыми или очень светлыми, с акцентом на число и разрез.
- Внутри карточки использовать короткие пары `подпись -> значение`, а не длинные предложения.
- Для статусов использовать малые badges: `Нет сети`, `Обновление`, `Нет товара`, `Ошибка`.
- Списки товаров и курьеров важнее декоративных плашек. Read-only позиции без drill-down показывать как современный table/list, а не как карточки с ложной кликабельностью.
- Иконка может помогать сканированию, но не заменяет подпись и число.
- На мобильном не допускать горизонтального переполнения длинных названий товара, клиента или курьера.

## Риски И Rollback

- Риск: слишком много запросов на commercial home.  
  Rollback: убрать `listClients` с home, оставить клиентов в нижней навигации.

- Риск: карточки станут перегруженными на маленьком экране.  
  Rollback: оставить в карточке 2-3 значения, остальное перенести в список ниже.

- Риск: появится общий компонент, который начнет тянуть доменную логику.  
  Rollback: вернуть JSX в ролевой/feature-компонент, оставить только CSS-классы.

- Риск: после перестановки потеряется invalidation/refetch.  
  Rollback: не трогать sale/load feature-компоненты в этом плане, role home только открывает вкладку.
