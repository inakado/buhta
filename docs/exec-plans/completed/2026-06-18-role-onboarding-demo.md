# Role Onboarding Demo

Статус: `Completed`
Дата: `2026-06-18`
Завершено: `2026-06-20`

## 1. Цель

Сделать ролевой интерактивный онбординг, который показывает пользователю принцип работы его роли через реальные экраны CRM, заполненные моковыми данными.

Пользователь должен быстро понять:

- что означают основные вкладки и кнопки его роли;
- какой рабочий процесс от него ожидается;
- какие действия он может выполнить сам;
- какие ограничения роли нормальны и к кому относится недоступное действие.

Онбординг должен быть отдельным демо-разделом во вкладке `Еще`, без изменения рабочих данных и без обучающего слоя для администратора.

## 2. Scope

- Добавить role-specific entry во вкладку `Еще` для ролей:
  - `commercial_manager` — `Как работают продажи`;
  - `production_manager` — `Как работает цех`.
- Не добавлять онбординг для `admin`.
- Сделать интерактивный demo flow с перелистыванием страниц/шагов.
- На каждом шаге показывать один фрагмент настоящего интерфейса роли с примерными данными, выделенный зеленой рамкой без заливки.
- Объяснять назначение фрагмента коротким текстом рядом с интерфейсом.
- Показывать ролевые ограничения явно, чтобы пользователь не искал недоступные функции:
  - заведующему производством объяснить, что новое сырье, тара и шаблоны продукции заводятся директором или администратором;
  - коммерческому руководителю объяснить, что курьерские балансы являются контрольным экраном, а загрузку и возврат оформляет курьер.
- Использовать только моковые данные внутри demo mode: сырье, тара, продукция, клиенты, курьеры, распределители, задачи и суммы.
- Не записывать действия из онбординга в API, БД, audit/history и реальные балансы.
- Сохранить mobile-first UX, совместимый с текущим PWA shell и нижней навигацией.

Фактическая поставка закрывает две роли, которые требовались для первого пользовательского прогона: коммерческий руководитель и заведующий производством. Онбординг Директора, работника распределителя и курьера не входит в завершенный scope этого плана и может быть оформлен отдельной инициативой после обратной связи.

## 3. Out Of Scope

- Онбординг администратора.
- Онбординг Директора, работника распределителя и курьера в рамках этой закрытой поставки.
- Кнопка `Понятно` на каждом шаге. Управление: перелистывание вперед/назад и закрытие.
- Полноценная sandbox-сессия с настоящими API-запросами.
- Видео, анимационные туры поверх реального рабочего экрана и forced first-login wizard.
- Переработка прав, ролей, доменной модели или API.
- Хранение прогресса онбординга на backend.
- Демо для десктопной админ-панели как отдельный продуктовый режим.

## 4. UX Решения

- Entry point: рабочая строка во вкладке `Еще` у `production_manager` и `commercial_manager`, без учебных служебных ярлыков внутри demo screen.
- Pattern: отдельный экран demo mode, а не modal поверх рабочей формы. Так пользователь не путает обучение с реальной операцией.
- Навигация по обучению:
  - swipe/перелистывание шагов;
  - кнопки `Назад`, `Вперед`;
  - без отдельного крестика закрытия, возврат идет через нижнюю навигацию;
  - индикатор `1 / N`;
  - controls перелистывания стоят отдельным footer-блоком над нижней навигацией и не скроллятся вместе с контентом шага;
  - без отдельной кнопки `Понятно`.
- Каждый шаг состоит из:
  - одного ключевого фрагмента интерфейса, выделенного рамкой;
  - для операции можно показывать соседние фрагменты точки входа и заполненной формы, например отдельная кнопка `Добавить сырье`/`Добавить тару`/`Выпустить`/`Передать` и ниже форма операции без повторяющего заголовка;
  - короткого объяснения в отдельной текстовой карточке без заголовков и жирных акцентов: что это, зачем нужно, что будет после действия;
  - controls перелистывания;
  - последний шаг про ограничения роли должен быть чисто информационным, без отдельного visual-фрагмента, формы и submit-кнопки.
- Язык: конкретный и операционный, без маркетинговых обещаний и длинных инструкций.
- Моковые данные должны выглядеть как рабочая Бухта:
  - продукция: `Икра А`, `Икра B`, `Филе гребешка`;
  - сырье: `Сырье лосося`, `Соль`, `Масло`;
  - тара: `Банка 250 мл`, `Крышка`, `Этикетка`;
  - клиенты: `Петр Петров`, `Мария Иванова`;
  - места: `Портовая 1`, `Цех`;
  - операции: продажа, выпуск, передача на распределитель, задача производству.
- Визуально demo screen должен быть похож на настоящий интерфейс, но отделен от работы через вход из `Еще`, примерные данные, заблокированные submit-действия или имитацию submit без API. Видимые ярлыки `Обучение`, `Демо`, `Моковые данные` и `Фокус` не использовать.

## 5. Ролевые Сценарии

### Коммерческий руководитель

- Клиенты: поиск, добавление, редактирование.
- Продажа с распределителя: клиент, продукция, количество, тип оплаты, result layer.
- Курьеры: зачем нужны балансы и контроль товара/наличных.
- Задачи производству: как поставить задачу и где она появится у производства.
- Ограничение: загрузку и возврат курьер оформляет сам; отдельный финальный шаг про сырье, тару и шаблоны в коммерческом обучении не показывается.

### Заведующий производством

- Производственный дом: сырье, тара, выпуск, передача.
- Поступление сырья/тары: как фиксируется приход.
- Выпуск продукции: как расходуются сырье и тара, что появляется после `Записано`.
- Передача на распределитель: как продукция уходит из цеха.
- Задачи: где появляются поручения от коммерческого руководителя.
- Ограничение: новые виды сырья, тары и шаблоны продукции заводит директор или администратор; если нужного нет в списке, это не ошибка экрана.

## 8. API/Contracts Changes

API и backend менять не планируется.

Demo mode должен быть полностью frontend-only:

- данные берутся из статических моков;
- submit-действия имитируются локально;
- query invalidation не вызывается;
- операция не попадает в audit/history;
- backend permissions остаются источником истины для реальной CRM.

## 9. Шаги Реализации

Реализация выполнена для `production_manager` и `commercial_manager`: отдельный frontend-only demo screen без крестика, с зеленой рамкой без заливки, текстовой карточкой и фиксированными controls над нижней навигацией.

1. Спроектировать shared onboarding model:
   - список ролей текущей поставки: `production_manager` и `commercial_manager`;
   - набор шагов по роли;
   - тип шага: фрагмент интерфейса и поясняющий текст.
2. Добавить entry point в `Еще`:
   - role-specific entry label для `production_manager` и `commercial_manager`;
   - без строки у администратора и прочих ролей текущей поставки;
   - возврат из обучения обратно в `Еще`.
3. Реализовать `RoleOnboardingHome`:
   - перелистывание вперед/назад;
   - закрытие;
   - индикатор шага;
   - mobile layout без вложенных карточек.
4. Собрать mock UI fragments:
   - переиспользовать текущие визуальные классы и компоненты там, где это не тянет реальные queries/mutations;
   - для сложных форм создать read-only demo fragments с тем же visual language;
   - заполнить моковыми данными по сырью, таре, продукции, клиентам и операциям.
5. Описать ограничения ролей:
   - отдельные шаги с недоступными действиями;
   - короткое объяснение, какая роль выполняет действие.
6. Защитить от случайной записи:
   - все demo buttons внутри onboarding не вызывают API;
   - submit-like кнопки показывают локальный demo result или переводят на следующий шаг.
7. Обновить тесты и документацию:
   - web tests на наличие/отсутствие role-specific onboarding по ролям;
   - тесты перелистывания;
   - тест, что admin не видит onboarding;
   - `docs/FRONTEND.md` с финальным pattern.

## 11. Риски И Rollback

- Risk: пользователь спутает demo mode с реальной работой.
  - Mitigation: отдельный экран из `Еще`, примерные имена и отсутствие API submit.
- Risk: копия интерфейса быстро устареет при изменении рабочих экранов.
  - Mitigation: переиспользовать CSS/components, а не рисовать отдельный стиль; держать onboarding fragments компактными и роль-ориентированными.
- Risk: слишком много текста снова перегрузит пользователя.
  - Mitigation: один фрагмент интерфейса и одна мысль на шаг; длинные объяснения заменить конкретным ограничением роли.
- Risk: добавление обучения в `Еще` утяжелит навигацию.
  - Mitigation: показывать одну рабочую строку во вкладке `Еще`; не добавлять новый bottom tab.
- Rollback: удалить entry point, shared onboarding modules и тесты; backend/data cleanup не требуется, потому что demo mode не пишет данные.
