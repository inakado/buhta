# FRONTEND

Frontend notes проекта «Бухта».

Статус: `Draft`. Фиксирует базовое направление frontend: mobile-first PWA. Не фиксирует финальную дизайн-систему до появления первых экранов.

## 1. Назначение

Документ нужен для правил frontend-реализации после scaffold:

- mobile-first UX;
- PWA/app-like поведение;
- performance budget;
- структура web-приложения;
- server-state conventions;
- формы и валидация;
- UI primitives;
- тестирование пользовательских сценариев.

## 2. Product Direction

Большая часть ежедневной работы пользователей должна выполняться с мобильных устройств.

Основной интерфейс проектируется как мобильное приложение:

- заведующий производством, работник распределителя, курьер и коммерческий руководитель должны быстро выполнять ежедневные операции с телефона;
- компьютер и планшет используются преимущественно для аудита, просмотра истории, статистики и более широких отчетных экранов;
- desktop-версия не является основной точкой проектирования для операционных сценариев.

Выбранное направление — `PWA`:

- приложение должно устанавливаться на экран устройства;
- в установленном режиме оно должно ощущаться как нативное приложение;
- PWA используется для app-like UX, скорости доступа и повторного ежедневного использования;
- полноценный offline-first режим в v1 не делаем.

## 3. Уже принятые правила

- Frontend не является источником доменных инвариантов.
- Server-state держим в выделенном слое.
- Финальные проверки прав, остатков, денег, скидок и статусов делает backend.
- UI должен быть рабочим инструментом для внутренней команды, без маркетинговой подачи.
- Базовый UX — mobile-first.
- Операционные сценарии проектируются под 2-3 основных касания до действия, если бизнес-правило не требует дополнительного подтверждения.
- Интерфейс должен быть стильным, но быстрым, читаемым и не перегруженным декоративными элементами.
- Критичные write-операции выполняются только online.
- Offline в v1 допускается только для безопасного кеширования shell/static/read-only данных, если это не создает конфликтов остатков и денег.
- Сложные интерактивные UI primitives строим на Radix UI, если для pattern есть подходящий primitive: dialog, popover, select, combobox-like picker, tabs/segmented controls и похожие составные controls. В проект добавляется только нужный `@radix-ui/react-*` пакет, а визуальная оболочка всегда остается нашей дизайн-системой; готовые темы и чужие visual kits не подключать.
- Временные CSS fallback-классы и inline styles не считать финальным решением. Если dev-server/HMR не подхватил новые классы, сначала диагностировать stale CSS, hard reload или перезапуск dev-server; устойчивые visual patterns фиксировать в `globals.css` и предметных class names.
- Disabled action tiles не использовать как placeholder будущих фич. Плитка действия должна вести к реальному flow.
- Некликабельные read-only строки товара, клиента, курьера или справочника не должны выглядеть как карточки с drill-down. Для них использовать плотный list/table pattern с разделителями.
- Декоративные `summary-card` / `entity-card` не являются базовым primitive проекта. Ролевая карточка допустима только как предметный overview конкретной роли, а form/card рамка допустима только для реальной формы или режима редактирования.

## 4. Mobile UX Rules

Операционные mobile flows должны быть короткими и предсказуемыми:

- основные действия доступны с первого экрана роли;
- touch targets достаточно крупные для работы одной рукой;
- нижняя навигация или аналогичный mobile-native pattern предпочтительнее desktop-sidebar для ежедневных операций;
- формы короткие, с понятными default values и быстрым вводом количества/способа оплаты;
- таблицы на мобильном не являются базовым pattern: использовать карточки, списки, summary rows и drill-down;
- ошибки объясняют, что именно мешает действию: нет остатка, нет прав, нет соединения, операция уже выполнена;
- loading states должны быть явными и короткими, без блокировки всего интерфейса без причины;
- опасные действия требуют подтверждения, но обычные частые операции не должны превращаться в длинный wizard.

## 5. Online-only Operations

В v1 все операции, меняющие состояние, выполняются только при доступном соединении с backend:

- продажа;
- загрузка продукции курьером;
- сгрузка продукции и наличных курьером;
- выпуск продукции;
- перемещение продукции на распределитель;
- списание наличных;
- назначение или использование заранее заданного дисконта;
- изменение статусов задач/уведомлений.

Причина: деньги, остатки и история действий должны оставаться согласованными. Offline queue для таких операций может быть отдельным будущим решением, но не входит в v1.

UI должен явно показывать состояние соединения. Если соединения нет, write-действие недоступно или завершается понятной ошибкой без локального “успешного” состояния.

## 6. Performance Rules

Скорость важна как часть продукта, а не только как техническая метрика.

Правила:

- стартовый экран роли должен открываться быстро на мобильной сети;
- не грузить тяжелые desktop/audit экраны в операционные mobile flows;
- client components использовать только там, где нужна интерактивность;
- server-state не дублировать в произвольный локальный state;
- изображения, иконки и шрифты не должны тормозить первый экран;
- UI не должен ждать лишних запросов, если данные уже есть в server-state cache;
- после действия пользователь должен быстро получить подтверждение результата или понятную ошибку;
- оптимизация не должна ухудшать читаемость и качество интерфейса.

## 7. Desktop And Tablet Scope

Большой экран используется для задач, где он реально дает преимущество:

- аудит истории действий;
- статистика и отчеты;
- просмотр продаж, остатков, денег и дисконтов в разрезах;
- административные настройки;
- расследование ошибок и сверка операций.

Desktop/tablet интерфейс может быть более плотным, с таблицами и фильтрами. Он не должен диктовать структуру мобильных операционных экранов.

## 8. PWA Scope For v1

В v1 PWA означает:

- web app manifest;
- app icons;
- standalone display mode;
- mobile theme/status colors;
- безопасное кеширование shell/static assets;
- быстрая повторная загрузка приложения.

PWA/mobile foundation реализуется в ранней части v1, а не как финальная полировка после отчетов. Минимальный ранний контур:

- app-like shell;
- role home screens;
- mobile navigation;
- explicit offline write blocking;
- базовые loading/error states для write-действий.

Не входит в v1:

- offline queue для продаж и денежных операций;
- автоматическое разрешение конфликтов остатков;
- background sync для write-операций;
- reliance на push notifications как обязательный канал работы.

## 9. Design Direction Draft

Этот раздел фиксирует направление для первых экранов, но не финальную дизайн-систему.

Визуальный ориентир: чистый mobile app, светлая поверхность, графитово-черные action/navigation элементы, зеленый акцент и мягкие lime/green summary states. Не копировать чужие экраны, тексты, изображения и композицию один-в-один.

Черновая палитра:

| Token | Hex | Использование |
| --- | --- | --- |
| `brand.green` | `#4AB855` | основной зеленый акцент, активные состояния, progress/status |
| `brand.lime` | `#E8FBCB` | мягкие gradient cards, вторичный акцент, light success background |
| `base.black` | `#000000` | нижняя навигация, primary action, контрастные статусы, основной текст на светлом фоне |
| `base.white` | `#FFFFFF` | основные поверхности, карточки, текст на темном фоне |

Для фона приложения можно использовать очень светлый серо-зеленый оттенок после подбора на реальных экранах. Пока он не фиксируется как обязательный token.

Правило сессии для app shell:

- кнопка выхода не должна находиться на основном рабочем экране роли;
- выход размещается в настройках/профиле;
- после нажатия `Выйти` UI сразу сбрасывает локальное состояние сессии и возвращает пользователя на экран ввода логина и пароля, а серверный sign-out очищает cookie следом.

Визуальные правила для первых экранов:

- mobile-first layout с ощущением установленного приложения;
- светлый фон страницы и белые основные поверхности;
- черный или почти черный цвет для нижней навигации, primary action и контрастных статусов;
- зеленый акцент для успешных, активных и операционных состояний;
- lime/green gradients использовать точечно: summary cards, balance cards, progress/status cards;
- не превращать весь интерфейс в одну зеленую палитру: обязательно балансировать белым, серым и черным;
- типографика нейтральная grotesk/system stack; если в проекте уже подключен `Inter`, использовать его как практичный стартовый аналог;
- декоративность не должна ухудшать скорость чтения остатков, денег, ролей и статусов;
- функциональные карточки должны оставаться компактными; крупные радиусы и градиенты допустимы только для акцентных summary-блоков после UI review;
- иконки в navigation/actions предпочтительнее длинных текстовых кнопок, но текстовые подписи сохранять там, где это снижает риск ошибки.

Базовые семейства компонентов, которые стоит выделить при реализации shell:

- `AppShell` — mobile-safe wrapper приложения;
- `HeaderBar` — роль, имя пользователя, статус соединения, быстрые действия;
- `BottomNav` — нижняя навигация роли;
- role overview blocks — баланс, остатки, деньги, ключевой статус конкретной роли без универсального builder;
- `ActionTile` — короткое действие на главном экране роли;
- read-only list/table rows — товары, клиенты, справочники, пользователи и операции без ложной кликабельности;
- `StatusBadge` — роль, статус операции, online/offline, delivered/blocked/error;
- `ActionFooter` — закрепленные действия формы;
- `FormPanel` — компактная форма бизнес-операции;
- `AuditEventRow` — запись истории действий.

Первые frontend screens должны показать, как эти компоненты работают вместе на телефоне. После этого раздел нужно либо превратить в стабильные UI conventions, либо заменить отдельным design system документом.

Фактически реализованный курьерский контур использует те же mobile patterns:

- home курьера показывает товарный баланс и наличный баланс курьера;
- экран `Загрузка` оформляет self-load с распределителя на баланс курьера и открывается только плиткой действия на home;
- экран `Продажа` оформляет продажу со своего `CourierProductBalance`, поддерживает поиск/создание клиента, выбор cash/cashless и success notice и открывается только плиткой действия на home;
- экран `Возврат` оформляет возврат одной или нескольких товарных строк и/или наличных на выбранное место и открывается только плиткой действия на home;
- экран `Курьеры` у коммерческого руководителя и Директора показывает read-only товарные и наличные балансы курьеров;
- admin UI для courier sale/load/unload намеренно не добавлен, хотя backend поддерживает admin flow для write-команд с явным курьером.

Фактический priced stock / discount UI:

- `DistributorStockList` показывает главным числом фактическую цену строки `unitPriceCents`;
- если строка дисконтирована, компактная иконка снижения показывается рядом с названием продукции, без видимого текстового badge, а базовая цена партии выводится вторичной зачеркнутой строкой;
- в sale product select для распределителя и курьера дисконтированные priced rows помечаются той же иконкой рядом с названием продукции;
- sale combobox для клиента и продукции открывается по focus и до ввода показывает первые 3 доступных варианта;
- у Директора во вкладке `Распределитель` есть inline action `Снизить цену` рядом с товарной строкой;
- форма дисконта открывается от конкретной строки остатка и отправляет только `distributorProductBalanceId`, количество, новую цену и optional комментарий;
- submit дисконта disabled при offline, некорректном количестве, цене `<= 0`, цене не ниже текущей или количестве сверх остатка;
- работник распределителя, коммерческий руководитель и курьер видят priced stock как свойство товара, но не могут изменить цену в sale/load/unload flow;
- sale/load/unload screens считают totals по `unitPriceCents` выбранной строки и не отправляют цену продажи на backend.

Фактический commercial manager home:

- роль собирается в `apps/web/src/roles/commercial-manager/CommercialManagerHome.tsx` через `RoleHomeRouter`;
- home имеет заголовок `Продажи`, одну верхнюю карточку `Остаток распределителя` с количеством, стоимостью, числом позиций и наличными, затем primary action tile `Продать`;
- верхняя карточка commercial home использует цельную lime/green поверхность без вложенных белых/серых карточек; внутренние значения разделяются типографикой и мягкими зелеными линиями;
- подробный список остатков не показывается на home коммерческого руководителя: он открывается отдельным нижним пунктом `Остатки`;
- `Клиенты` и `Курьеры` не дублируются плитками на home, потому что эти переходы уже есть в нижней навигации;
- `Продажа` не показывается в нижней навигации: переход в продажу выполняет только плитка действия `Продать`;
- `История` показывается отдельной нижней вкладкой и содержит compact backend-backed список последних продаж из `/distributor/sales/recent` с реальным статусом отмены;
- блоки `Продажи сегодня`, fake revenue и декоративную статистику продаж не включать без надежного backend/read model.

Фактический distributor worker home:

- роль собирается в `apps/web/src/roles/distributor-worker/DistributorWorkerHome.tsx` через `RoleHomeRouter`;
- home использует общий доменный `features/distributor/DistributorHomeOverview.tsx`, потому что commercial manager и distributor worker показывают один distributor inventory/cash контур;
- home имеет заголовок `Распределитель`, один компактный рабочий блок `Стоимость` и `Наличные`, primary action tile `Продать` и inline table/list `Продукция`;
- `Продажа` не показывается в нижней навигации: переход в продажу выполняет только плитка действия `Продать`;
- `История` показывается отдельной нижней вкладкой, а не плиткой рядом с `Продать`;
- плитки `Остатки`, `Клиенты`, `Курьеры` не добавлять: продукция уже на home, клиенты есть в нижней навигации, курьеры не входят в права worker.

Фактический courier home:

- роль собирается в `apps/web/src/roles/courier/CourierHome.tsx` через `RoleHomeRouter`;
- home использует доменный `features/courier/CourierHomeOverview.tsx`;
- home имеет заголовок `Мой баланс`, один компактный рабочий блок `Стоимость` и `Наличные`, action tiles `Продать` и `Загрузить`, inline table/list `Продукция`;
- home имеет action tiles `Продать`, `Загрузить` и `Вернуть`;
- `Продажа`, `Загрузка` и `Возврат` не показываются в нижней навигации: переходы выполняют только плитки действий на home;
- disabled/placeholder action tiles не используются: плитка `Вернуть` появилась только после backend flow.

Фактический courier unload flow:

- `CourierUnloadHome` остается feature-компонентом, а не частью role home: role screen открывает действие `Вернуть`;
- options endpoint возвращает активные распределители, товарные строки курьера и cash balance;
- если активный распределитель один, он выбран по умолчанию;
- форма по умолчанию подставляет весь доступный товар и всю сумму наличных;
- в UI экран называется `Возврат`, плитка входа называется `Вернуть`, блок выбора называется `Куда вернуть`, дополнительный видимый label над select не используется, товарные строки используют действие `Вернуть`, а submit — `Записать`;
- quantity `0` означает "не возвращать строку", но нулевые строки не отправляются в request;
- cash-only и product-only сценарии поддерживаются;
- после успешного возврата показывается global success notice `Возврат записан` и пользователь возвращается на home;
- submit disabled при offline, отсутствии распределителя, пустой операции, некорректном количестве или сумме наличных сверх баланса;
- если submit disabled, форма показывает короткую причину блокировки рядом с кнопкой;
- query invalidation после возврата обновляет courier balances, unload options, sale/load options и distributor inventory/cash projections.

Фактический courier sale flow:

- `CourierSaleHome` остается feature-компонентом, а не частью role home: role screen только открывает действие `Продать`;
- структура сверена с `13-sale-flow.html`: searchable client combobox/create client, выбор продукции с баланса курьера, количество, payment segmented control и operation summary;
- клиент выбирается через один searchable combobox: пользователь вводит имя/телефон в поле `Клиент`, после ввода видит результаты и выбирает клиента из списка, а выбранного клиента можно очистить кнопкой `Очистить клиента`;
- после выбора клиента кнопка `Новый клиент` скрывается; после очистки клиента создание клиента снова доступно;
- экран продажи не использует верхнюю decorative summary-card: пользователь идет сверху вниз от клиента и продукции к оплате, а сумма показывается только в нижнем плоском operation summary;
- operation summary показывает клиента, товар, количество, способ оплаты и сумму; фразы про изменение наличных не выводятся;
- query invalidation после продажи и создания клиента сохраняется в feature-компоненте;
- дополнительную sticky action abstraction пока не вводить: достаточно scroll padding, чтобы submit не попадал под bottom nav.

Фактические read-only/list rules после cleanup:

- `RoleHomeRouter` не содержит общий fallback-home с disabled action tiles; неизвестный/будущий раздел показывается плоским `placeholder-panel`;
- справочники начинаются с обычного заголовка и tabs, без decorative summary-card;
- обычные строки справочников используют `catalog-list-row`, а форма редактирования остается `form-panel edit-card`;
- клиенты отображаются через `client-list-table` / `client-list-row`, а не `entity-card`;
- собственный блок наличных курьера использует `flat-balance-row`;
- `Продукция в цеху` у заведующего производством использует `inventory-table-*`, как история и read-only остатки.

Фактический director home:

- роль собирается в `apps/web/src/roles/director/DirectorHome.tsx` через `RoleHomeRouter`;
- home показывает только компактную сводку контроля: общий товар в обороте, товарную стоимость, наличные и два контура `Распределитель`/`Курьеры`;
- детальные списки не выводятся на home: распределитель открывается отдельной нижней вкладкой `Распределитель`, курьеры — вкладкой `Курьеры`;
- fake-блоки `Продажи сегодня`, `Последние операции`, fake revenue/report numbers не добавлять до backend/read model;
- управленческие действия `Назначить дисконт`, `Отчеты`, `История` не показываются как disabled tiles на home, пока backend flow не реализован;
- bottom navigation директора не показывает `Продажа`, потому что у роли нет write sale action.

Фактический admin users screen:

- `apps/web/src/roles/admin/AdminHome.tsx` является тонким wrapper и подключает `features/users/AdminUsersHome.tsx`;
- экран начинается обычным заголовком `Пользователи` с количеством сотрудников и компактной кнопкой `Новый`, без decorative summary-card;
- форма создания сотрудника не открыта постоянно: она раскрывается только по действию `Новый` и закрывается после успешного создания;
- временный пароль показывается отдельным compact notice с возможностью копирования и закрытия;
- сотрудники отображаются как плотный access-list, а не как `entity-card`: строка показывает имя, login, role select и сброс пароля;
- сброс пароля требует локального подтверждения в строке и после успеха обновляет `["users"]`;
- роль admin не получает операционные sale/load UI, хотя backend может поддерживать часть этих flows.

Фактический operation history screen:

- `apps/web/src/features/operations/OperationHistoryHome.tsx` является read-only feature-компонентом для `admin` и `director`;
- нижняя навигация показывает вкладку `История` только при permission `operation.history.read`;
- admin и директор используют одно название `История`, без технического слова `Аудит` в UI;
- экран загружает список через `GET /operations/history`, а варианты фильтров через отдельный `GET /operations/history/options`;
- фильтры остаются компактным блоком сверху: период, событие `operationType`, пользователь, роль и источник;
- список использует плотные строки с разделителями, а не декоративные карточки: главное действие, actor/роль/источник/дата, сумма и количество;
- кнопка `Показать еще` использует backend cursor и не пытается разбирать cursor на frontend;
- details открываются через Radix Dialog и пока отображаются универсальным structured/key-value layout с короткой summary-зоной;
- кастомные секции под отдельные operation types отложены на следующий мини-этап;
- frontend не выполняет security redaction самостоятельно как источник истины: backend уже маскирует details, UI только форматирует полученные значения.

## 10. Component Architecture

Frontend структура должна помогать быстро находить код и не превращать role screens в большие файлы.

Принципы:

- route-level `page.tsx` должен быть тонким wrapper, а не местом бизнес-логики;
- role home component должен собирать экран из секций и feature-компонентов;
- бизнес-функции, которые переиспользуются ролями, живут в `features/*`, а не внутри конкретной роли;
- роль подключает feature в нужном контексте, но не копирует форму или логику;
- общие визуальные primitives живут в `ui/*`;
- `shared` package не использовать для React-компонентов.

Целевая структура web:


Пример переиспользования:

- продажа нужна курьеру, работнику распределителя и коммерческому руководителю;
- UI продажи должен жить в `features/sales/components`;
- `CourierHome`, `DistributorWorkerHome` и `CommercialManagerHome` подключают один `SaleEntryPanel` с разным источником/режимом;
- backend финально проверяет права и источник продажи, frontend только подстраивает форму под контекст роли.

Граница ответственности:

- `roles/*` — стартовые экраны ролей и композиция секций;
- `features/*` — бизнес-сценарии и переиспользуемые feature-компоненты;
- `app-shell/*` — оболочка приложения, навигация, online/offline state;
- `auth/*` — login/logout/current actor frontend flow;
- `ui/*` — визуальные primitives без доменных правил.

Текущий catalog foundation реализован как общий feature в `apps/web/src/features/catalog/CatalogHome.tsx` и подключен через mobile shell для пользователей с permission `catalog.manage`. Экран справочников не принадлежит только роли администратора: администратор и директор используют один feature, остальные роли не видят этот раздел в навигации. Write-действия на экране справочников отключаются при offline state.

Текущий production baseline реализован как feature в `apps/web/src/features/production/ProductionHome.tsx` и подключен через `RoleHomeRouter` для роли `production_manager`.

Фактическая нижняя навигация заведующего производством:

- `Главная` — карточка `Цех` с показателями `Продукция`, `Сырье` и `Тара`, быстрые действия без встроенных списков;
- `Распределитель` — read-only экран `На распределителе`;
- `Уведомления` — рабочий экран `Задачи производству` со списком задач, polling 30 секунд и действием `Выполнено` для новых задач;
- `История` — список последних выпущенных партий;
- `Профиль` — настройки и выход.

Фактический notification flow:

- `NotificationsHome` используется для production/commercial экранов и остается feature-компонентом;
- commercial manager открывает экран задач через home action `Уведомить`, видит общий список задач производству и форму `Что передать производству`;
- production manager открывает тот же список через вкладку `Уведомления`, но без формы создания;
- production manager видит компактный счетчик новых задач на пункте нижней навигации `Уведомления`;
- production manager может отметить новую задачу кнопкой `Выполнено`;
- список обновляется через React Query polling каждые 30 секунд, refetch on focus и invalidation после create/complete;
- write-действия disabled offline и показывают короткую причину блокировки.

Экран `Настройки` не использует декоративную summary-карточку: он показывает обычный заголовок, компактный блок профиля с ролью и логином, затем кнопку выхода.

Нижняя навигация визуально icon-only, но каждый пункт сохраняет `aria-label` с названием раздела.

Визуальный ориентир production home:

- основной шрифт — Helvetica;
- веса шрифта задаются через CSS-токены: `400` regular/labels, `500` emphasis, `600` только как резерв для редких сильных акцентов;
- карточки быстрых действий используют мягкий фон `#eff1ee` без рамки;
- карточка `Цех` на главной использует светлый lime/green фон, крупный показатель `Продукция` и две внутренние рабочие зоны `Сырье`/`Тара` без вложенных белых карточек;
- черный цвет остается для нижней навигации и основных submit-кнопок, а не для плиток быстрых действий;
- плитки действий крупно скруглены и остаются похожими на нативные mobile actions.
- иконки внутри агрегатов и списков сидят в мягких зеленых квадратах с согласованным скруглением, без острых внутренних углов.
- production iconography не дублирует смысловые уровни: готовая продукция — `PackageCheck`, сырье — лаконичный `FishSymbol`, тара — `Package`, выпуск — `Factory`, добавление тары — `PackagePlus`, добавление сырья — нейтральный `CirclePlus`.

Кликабельные зоны карточки `Цех` открывают отдельные экраны с кнопкой `Назад`:

- `Продукция` — список готовых партий в цеху;
- `Сырье` — виды сырья с ненулевым остатком и количество по каждому виду;
- `Тара` — виды тары с ненулевым остатком и количество по каждому виду.

Быстрые действия на главной производства открывают отдельные короткие экраны с кнопкой `Назад`:

- `Выпустить` — только форма выпуска продукции;
- `Передать` — форма `На распределитель`, перемещение готовой продукции из цеха на активный распределитель;
- `Добавить сырье` — только форма прихода сырья;
- `Добавить тару` — только форма прихода тары.

Цена шаблона продукции вводится в справочниках в рублях, но в API отправляется как `priceCents`. Production release использует активные шаблоны с ценой и показывает цену как snapshot будущей партии.

Форма `Выпуск продукции` после выбора шаблона показывает компактную строку доступности: сколько сырья и тары сейчас есть в цеху для выбранного шаблона. Если выбранному шаблону явно не хватает сырья или тары, форма показывает warning до submit и блокирует кнопку выпуска. Остаток после выпуска в этой форме не показывается, чтобы не перегружать рабочий экран.

Экран “Продукция в цеху” показывает доступный остаток готовой продукции из `GET /production/workshop-product-balances`, а не все выпущенные партии. История по-прежнему показывает последние выпуски через `GET /production/product-batches`; read-only выпуски отображаются как плотный table/list без карточек, потому что строка истории не открывает detail-screen.

Форма `На распределитель` использует `GET /production/transfer-options`, показывает доступное количество, цену партии и дату выпуска. Submit отправляет `POST /production/product-transfers`, блокируется offline и во время pending-состояния, а после успеха обновляет production summary, остатки цеха, transfer options и историю.

Production write actions используют легкое подтверждение результата: после успешного ответа backend экран возвращается на главную производства и показывает compact success notice снизу над навигацией. Spinner-first feedback не используется для быстрых операций; pending-состояние кнопок остается только защитой от двойного submit. Ошибки остаются inline в форме и не возвращают пользователя на главный экран.

Этот pattern сначала реализован только в production flow, но дальше должен расширяться на другие частые write-экраны CRM: короткое действие завершается возвратом в понятный контекст и ненавязчивым подтверждением результата. До появления общего toast/notice primitive не размазывать реализацию по всему приложению без отдельного решения; при расширении вынести общий компонент и оставить тексты подтверждений короткими.

Success notice не должен быть черным, потому что появляется над черной нижней навигацией. Базовый визуальный вариант — светлый `brand.lime` фон с темным текстом и короткой иконкой успеха, без кнопки закрытия и без action.

Production UI не читает `/catalog/*`, потому что `production_manager` не имеет права управления справочниками. Для форм производства используются read-only endpoints под `production.manage`.

Read-only экран остатков распределителя живет в `features/distributor/DistributorInventoryHome.tsx` и читает `GET /distributor/inventory`. Над списком он показывает один компактный lime/green блок `Товар`, `Стоимость`, `Наличные` в визуальном стиле карточки `Цех`, но без увеличения высоты и без больших summary-карточек; количество позиций остается справа в заголовке. Позиции ниже отображаются как плотный read-only table/list без карточек и иконок, чтобы не создавать ложное ожидание перехода внутрь позиции. Для ролей с `distributor.cash.read` этот экран дополнительно читает `GET /distributor/cash-balances` и показывает наличный баланс распределителя. Для ролей без этого permission cash balance не запрашивается и не отображается.

Для Директора во вкладке `Распределитель` этот же feature показывает компактное действие `Списать наличные`. Форма выбирает только активные распределители из `GET /distributor/cash-balances`; если активный распределитель один, он выбран автоматически. Комментарий optional. Operation summary показывает доступную сумму, сумму списания и остаток после списания. После успеха форма закрывается, cash balance обновляется, пользователь остается во вкладке `Распределитель` и видит `Наличные списаны`. Offline, нулевая сумма, сумма сверх cash balance и отсутствие активного распределителя блокируют submit.

Для `production_manager` экран подключен отдельной нижней вкладкой `Распределитель` и открывается с заголовком `На распределителе`, чтобы не смешивать production home с read-only остатками распределителя. Для остальных не-admin ролей с `distributor.stock.read` home показывает distributor inventory с заголовком `Остатки`. `admin` имеет backend read access, но отдельный admin UI для этого этапа не добавлен.

Экран клиентов живет в `features/clients/ClientsHome.tsx` и подключается отдельной вкладкой `Клиенты` для ролей с `client.read`. Экран начинается обычным заголовком `Клиенты` с количеством и компактным действием `Добавить клиента` справа сверху, без summary-карточки: основной поток здесь поиск и список, поэтому создание не должно разрывать search-results зону. Роли с `client.manage` видят создание и редактирование клиента, директор видит read-only список и поиск. `production_manager` вкладку клиентов не видит. После успешного создания или редактирования форма возвращает пользователя к списку и показывает compact success notice: `Клиент добавлен` или `Клиент обновлен`. Backend errors остаются inline в форме.

Экран продажи с распределителя живет в `features/sales/DistributorSaleHome.tsx` и открывается плиткой действия `Продать` для ролей с `distributor.sale.create`, кроме `admin`, для которого UI продажи пока не добавлен. Форма выбирает клиента через searchable combobox на базе `GET /clients?search=...`; при пустом поле combobox показывает первые 3 клиента. Форма может раскрыть компактную мини-форму `Новый клиент` и создать клиента через `POST /clients`, затем выбирает товарную строку из `GET /distributor/sale-options`; product select при пустом поле показывает первые 3 доступные товарные строки. После выбора клиента кнопка `Новый клиент` скрывается до очистки выбранного клиента. Продажа отправляется в `POST /distributor/sales` по `distributorProductBalanceId`, количеству и способу оплаты. Верхняя decorative summary-card и список последних продаж не используются: расчет суммы находится в нижнем плоском operation summary, а работа с уже созданными продажами вынесена во вкладку `История`. После успеха active tab становится `home`, форма очищается, обновляются inventory/sale-options/cash/recent queries и показывается success notice `Продажа записана`. Ошибки остаются inline, offline блокирует submit.

Экран `История продаж` для коммерческого руководителя и работника распределителя живет в `features/sales/SalesHistoryHome.tsx` и открывается отдельной нижней вкладкой `История`, чтобы `Продать` оставался единственным крупным действием на home. На экране не дублируется внутренний заголовок `Последние продажи`: один экранный заголовок задает контекст. Экран читает `GET /distributor/sales/recent?limit=10`. Строка показывает продукцию, клиента, количество, фактическую цену, сумму, способ оплаты, автора продажи и статус отмены; сумма и действие вынесены в отдельную правую колонку, чтобы не смешиваться с названием и деталями. Неотмененная продажа имеет вторичное действие `Отменить`; оно раскрывает inline-поле `Причина отмены` и отправляет `POST /distributor/sales/:saleId/cancel`. Причина обязательна, offline и pending блокируют submit. После успешной отмены пользователь остается в `Истории продаж`, видит local success notice `Продажа отменена`, обновляются inventory/sale-options/cash/recent queries; отмененная строка становится приглушенной и больше не показывает action.

Экран курьерского баланса живет в `features/courier/CourierBalanceHome.tsx` и читает `GET /courier/product-balances`. У курьера home показывает только собственный товарный баланс. У коммерческого руководителя и Директора вкладка `Курьеры` показывает read-only балансы всех курьеров: сверху один компактный блок `Товар`, `Стоимость`, `Наличные`, ниже секция `Курьеры`, где каждый курьер выглядит как нейтральная мини-ведомость без зеленой карточной подложки. В шапке блока имя слева, наличные справа, под именем количество позиций, без login handle. Ниже табличной зоной идут товарные позиции `Товар`, `Количество`, `Сумма`, внизу показывается `Итого товаром`. Курьеры разделяются отступом и тонкой нейтральной линией, а не цветной карточкой. Отдельный общий список `Товары`, большую summary-карточку, отдельную карточку наличных и два параллельных списка по курьерам на этом экране не использовать. `admin` имеет backend read access, но отдельный admin UI для этого этапа не добавлен.

Экран курьерской продажи живет в `features/courier/CourierSaleHome.tsx` и использует тот же sale form pattern, что распределитель: клиентский combobox с первыми 3 клиентами при пустом поле, product select с первыми 3 собственными товарными строками, segmented control для cash/cashless и плоский operation summary. Список последних продаж не показывается внутри формы продажи. У курьера есть отдельная нижняя вкладка `История`, которая использует общий `SalesHistoryHome`, читает `GET /courier/sales/recent?limit=10` и показывает только продажи текущего курьера. Отмена собственной продажи требует причину и отправляет `POST /courier/sales/:saleId/cancel`; после успеха обновляются courier product balances, sale-options, cash-balances и recent sales. Коммерческий руководитель и Директор не видят cancel action для курьерских продаж.

Экран загрузки курьера живет в `features/courier/CourierLoadHome.tsx` и открывается плиткой действия `Загрузить` только для роли `courier`. Форма выбирает строку товара из `GET /courier/load-options`, показывает доступное количество, цену партии и сумму загрузки, затем отправляет `POST /courier/loads` без `courierUserId`. Верхняя decorative summary-card не используется: расчет суммы находится в нижнем плоском operation summary. После успеха active tab становится `home`, обновляются courier balances, load options и distributor inventory, а пользователь видит success notice `Загрузка записана`. Ошибки остаются inline, offline и pending-состояние блокируют submit.
