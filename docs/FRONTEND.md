# FRONTEND

Frontend notes проекта «Бухта».

Статус: `Draft`. Фиксирует frontend-направление: mobile-first PWA и извлеченную Impeccable design system. Главная Директора и `Остатки Директора` остаются эталонами для контрольных/read-only поверхностей; остальные роли и админские экраны следуют тем же ledger, command, form and management standards.

## 1. Назначение

Документ нужен для правил frontend-реализации после scaffold:

- mobile-first UX;
- PWA/app-like поведение;
- performance budget;
- структура web-приложения;
- server-state conventions;
- формы и валидация;
- UI primitives;
- тестирование пользовательских сценариев.

## 2. Product Direction

Большая часть ежедневной работы пользователей должна выполняться с мобильных устройств.

Основной интерфейс проектируется как мобильное приложение:

- заведующий производством, работник распределителя, курьер и коммерческий руководитель должны быстро выполнять ежедневные операции с телефона;
- компьютер и планшет используются преимущественно для аудита, просмотра истории, статистики и более широких отчетных экранов;
- desktop-версия не является основной точкой проектирования для операционных сценариев.

Выбранное направление — `PWA`:

- приложение должно устанавливаться на экран устройства;
- в установленном режиме оно должно ощущаться как нативное приложение;
- PWA используется для app-like UX, скорости доступа и повторного ежедневного использования;
- полноценный offline-first режим в v1 не делаем.

## 3. Уже принятые правила

- Frontend не является источником доменных инвариантов.
- Server-state держим в выделенном слое.
- Финальные проверки прав, остатков, денег, скидок и статусов делает backend.
- UI должен быть рабочим инструментом для внутренней команды, без маркетинговой подачи.
- Базовый UX — mobile-first.
- Операционные сценарии проектируются под 2-3 основных касания до действия, если бизнес-правило не требует дополнительного подтверждения.
- Интерфейс должен быть стильным, но быстрым, читаемым и не перегруженным декоративными элементами.
- Критичные write-операции выполняются только online.
- Offline в v1 допускается только для безопасного кеширования shell/static/read-only данных, если это не создает конфликтов остатков и денег.
- Сложные интерактивные UI primitives строим на Radix UI, если для pattern есть подходящий primitive: dialog, popover, select, combobox-like picker, tabs/segmented controls и похожие составные controls. В проект добавляется только нужный `@radix-ui/react-*` пакет, а визуальная оболочка всегда остается нашей дизайн-системой; готовые темы и чужие visual kits не подключать.
- Временные CSS fallback-классы и inline styles не считать финальным решением. Если dev-server/HMR не подхватил новые классы, сначала диагностировать stale CSS, hard reload или перезапуск dev-server; устойчивые visual patterns фиксировать в `globals.css` и предметных class names.
- Disabled action tiles не использовать как placeholder будущих фич. Плитка действия должна вести к реальному flow.
- Некликабельные read-only строки товара, клиента, курьера или справочника не должны выглядеть как карточки с drill-down. Для них использовать плотный list/table pattern с разделителями.
- Декоративные `summary-card` / `entity-card` не являются базовым primitive проекта. Ролевая карточка допустима только как предметный overview конкретной роли, а form/card рамка допустима только для реальной формы или режима редактирования.

## 4. Mobile UX Rules

Операционные mobile flows должны быть короткими и предсказуемыми:

- основные действия доступны с первого экрана роли;
- touch targets достаточно крупные для работы одной рукой;
- нижняя навигация или аналогичный mobile-native pattern предпочтительнее desktop-sidebar для ежедневных операций;
- формы короткие, с понятными default values и быстрым вводом количества/способа оплаты;
- операционные mobile forms не должны превращаться в шумные таблицы; для контрольных/read-only экранов вроде директорских остатков и истории допустим compact table/list pattern с тонкими разделителями, короткими labels и tabular numbers;
- ошибки объясняют, что именно мешает действию: нет остатка, нет прав, нет соединения, операция уже выполнена;
- loading states должны быть явными и короткими, без блокировки всего интерфейса без причины;
- опасные действия требуют подтверждения, но обычные частые операции не должны превращаться в длинный wizard.
- успешные write-операции с деньгами или остатками не должны ограничиваться transient toast: после записи пользователь видит inline result layer с основными деталями операции и явными следующими действиями, например `Готово` и `Новая продажа`.

### Quantity input for finished product

Товарные операции с готовой продукцией проектируются под рабочий язык кг:

- формы выпуска, перемещения, продажи, дисконта, загрузки и возврата открываются в режиме ввода кг;
- рядом доступен компактный segmented switch `кг` / `шт`;
- переключатель не должен быть modal или отдельным шагом wizard;
- label поля меняется по режиму: `Количество, кг` или `Количество, шт`; для возврата допустим контекстный label `Вернуть, кг`;
- рядом с полем показывается расчет counterpart: при вводе кг — сколько будет списано штук, при вводе штук — какая масса нетто получится;
- доступный остаток показывается в обоих измерениях через `•`, например `20 кг • 100 шт`;
- в read-only stock tables не добавлять отдельную колонку кг: колонка `Количество` показывает массу и штуки одной строкой через `•`, а цену за штуку второй строкой, чтобы сохранить мобильную плотность;
- в директорских production summary и read-only таблицах выпуска использовать тот же компактный формат `кг • шт` внутри существующего поля количества, без добавления отдельной колонки и без второго ряда только ради массы;
- в таблице `Продукция в цеху` колонка называется `Остаток`: текущий остаток показывается основной строкой, а исходный объем партии показывается как `партия ...` только если партия уже частично передана;
- цена в UI остается `₽/шт`; не показывать и не подразумевать `₽/кг`;
- некратная масса показывает inline error рядом с полем и блокирует submit;
- frontend расчет нужен для UX, но backend остается источником истины и повторно валидирует `quantityInput`.

Визуально этот control относится к рабочим form primitives: плотный, читаемый, с привычными кнопками сегмента, без декоративной карточки вокруг карточки и без изменения layout при ошибке или смене режима.

## 5. Online-only Operations

В v1 все операции, меняющие состояние, выполняются только при доступном соединении с backend:

- продажа;
- загрузка продукции курьером;
- сгрузка продукции и наличных курьером;
- выпуск продукции;
- перемещение продукции на распределитель;
- списание наличных;
- назначение или использование заранее заданного дисконта;
- изменение статусов задач/уведомлений.

Причина: деньги, остатки и история действий должны оставаться согласованными. Offline queue для таких операций может быть отдельным будущим решением, но не входит в v1.

UI должен явно показывать состояние соединения. Если соединения нет, write-действие недоступно или завершается понятной ошибкой без локального “успешного” состояния.

## 6. Performance Rules

Скорость важна как часть продукта, а не только как техническая метрика.

Правила:

- стартовый экран роли должен открываться быстро на мобильной сети;
- не грузить тяжелые desktop/audit экраны в операционные mobile flows;
- client components использовать только там, где нужна интерактивность;
- server-state не дублировать в произвольный локальный state;
- изображения, иконки и шрифты не должны тормозить первый экран;
- UI не должен ждать лишних запросов, если данные уже есть в server-state cache;
- после действия пользователь должен быстро получить подтверждение результата или понятную ошибку;
- оптимизация не должна ухудшать читаемость и качество интерфейса.

## 7. Desktop And Tablet Scope

Большой экран используется для задач, где он реально дает преимущество:

- аудит истории действий;
- статистика и отчеты;
- просмотр продаж, остатков, денег и дисконтов в разрезах;
- административные настройки;
- расследование ошибок и сверка операций.

Desktop/tablet интерфейс может быть более плотным, с таблицами и фильтрами. Он не должен диктовать структуру мобильных операционных экранов.

## 8. PWA Scope For v1

В v1 PWA означает:

- web app manifest;
- app icons;
- standalone display mode;
- mobile theme/status colors;
- безопасное кеширование shell/static assets;
- быстрая повторная загрузка приложения.

PWA/mobile foundation реализуется в ранней части v1, а не как финальная полировка после отчетов. Минимальный ранний контур:

- app-like shell;
- role home screens;
- mobile navigation;
- explicit offline write blocking;
- базовые loading/error states для write-действий.

App icon использует Pearl Cove знак в `apps/web/app/icon.svg`, `/apple-touch-icon.png`, `/icons/icon-192.png` и `/icons/icon-512.png`. SVG остается favicon/source icon, PNG нужны для установленной PWA и iOS home-screen; старый знак с буквой `Б` не использовать. Auth loading screen использует компактный черный `loader-pearl-cove.svg` внутри светлого mobile shell, без старого точечного `loading-dot` и без текстовой подписи.

Manifest contract закреплен тестом: `display: standalone`, `start_url: /`, theme/background colors, maskable SVG icon and PNG install icons. Mobile shell reserves bottom space through `safe-area-inset-*` variables: app padding, bottom navigation and global success notices remain above the system home indicator in installed/mobile contexts. На телефонных viewport оболочка раскрывается на весь экран без серых внешних полей, border-radius и shadow, чтобы установленная PWA ощущалась как нативное приложение. Viewport фиксирует `initial-scale=1`, запрещает пользовательский page zoom и использует `viewport-fit=cover`; form controls на телефоне имеют computed font-size не ниже 16px, чтобы iOS не приближал страницу при фокусе на поле. Offline behavior остается read-only for data screens with explicit disabled write actions; offline write queue в v1 не добавляется.

Не входит в v1:

- offline queue для продаж и денежных операций;
- автоматическое разрешение конфликтов остатков;
- background sync для write-операций;
- reliance на push notifications как обязательный канал работы.

## 9. Design Direction Draft

Этот раздел фиксирует направление первых экранов и текущий переход к новой Impeccable design system. Постоянные visual rules переносятся в `DESIGN.md`, а этот документ хранит frontend conventions и ограничения применения.

Визуальный ориентир: чистый mobile app, светлая поверхность, графитово-черные action/navigation элементы, зеленый акцент и мягкие lime/green summary states. Не копировать чужие экраны, тексты, изображения и композицию один-в-один.

Черновая палитра:

| Token | Hex | Использование |
| --- | --- | --- |
| `brand.green` | `#4AB855` | основной зеленый акцент, активные состояния, progress/status |
| `brand.lime` | `#E8FBCB` | мягкие gradient cards, вторичный акцент, light success background |
| `base.black` | `#000000` | нижняя навигация, primary action, контрастные статусы, основной текст на светлом фоне |
| `base.white` | `#FFFFFF` | основные поверхности, карточки, текст на темном фоне |

Для фона приложения можно использовать очень светлый серо-зеленый оттенок после подбора на реальных экранах. Пока он не фиксируется как обязательный token.

Правило сессии для app shell:

- app shell не показывает постоянную верхнюю надпись роли (`Директор`, `Курьер` и т.п.) на каждом экране: роль остается в профиле, истории/audit details и account-блоках, а рабочий контекст задают нижняя навигация и заголовок текущего раздела;
- кнопка выхода не должна находиться на основном рабочем экране роли;
- выход размещается в настройках/профиле;
- после нажатия `Выйти` UI сразу сбрасывает локальное состояние сессии и возвращает пользователя на экран ввода логина и пароля, а серверный sign-out очищает cookie следом.

Визуальные правила для первых экранов:

- mobile-first layout с ощущением установленного приложения;
- светлый фон страницы и белые основные поверхности;
- черный или почти черный цвет для нижней навигации, primary action и контрастных статусов;
- зеленый акцент для успешных, активных и операционных состояний;
- lime/green gradients использовать точечно: summary cards, balance cards, progress/status cards;
- не превращать весь интерфейс в одну зеленую палитру: обязательно балансировать белым, серым и черным;
- типографика нейтральная grotesk/system stack; если в проекте уже подключен `Inter`, использовать его как практичный стартовый аналог;
- декоративность не должна ухудшать скорость чтения остатков, денег, ролей и статусов;
- рабочие панели должны оставаться компактными; крупные радиусы, градиенты и декоративные карточки не использовать для основных ролей без отдельного UI review;
- иконки в navigation/actions предпочтительнее длинных текстовых кнопок, но текстовые подписи сохранять там, где это снижает риск ошибки.

Базовые семейства компонентов, которые стоит выделить при реализации shell:

- `AppShell` — mobile-safe wrapper приложения;
- `BottomNav` — нижняя навигация роли;
- `OfflineBanner` / connection status — состояние сети только когда оно влияет на write-действия;
- role overview blocks — баланс, остатки, деньги, ключевой статус конкретной роли без универсального builder;
- command actions — короткие действия на главном экране роли через единый `production-command-button` rhythm;
- read-only list/table rows — товары, клиенты, справочники, пользователи и операции без ложной кликабельности;
- `StatusBadge` — роль, статус операции, online/offline, delivered/blocked/error;
- `ActionFooter` — закрепленные действия формы;
- `FormPanel` — компактная форма бизнес-операции;
- `AuditEventRow` — запись истории действий.

Первые frontend screens должны показать, как эти компоненты работают вместе на телефоне. После этого раздел нужно либо превратить в стабильные UI conventions, либо заменить отдельным design system документом.

Фактически реализованный курьерский контур использует те же mobile patterns:

- home курьера показывает товарный баланс и наличный баланс курьера;
- экран `Загрузка` оформляет self-load с распределителя на баланс курьера и открывается только command action на home;
- экран `Продажа` оформляет продажу со своего `CourierProductBalance`, поддерживает поиск/создание клиента, выбор cash/cashless и открывается только command action на home;
- после успешной курьерской продажи экран показывает inline result layer `Записано` с клиентом, компактной строкой `Продано`, способом оплаты, итогом и действиями `Готово` / `Новая продажа`; возврат на home происходит по `Готово`, а не автоматически сразу после записи;
- экран `Возврат` оформляет возврат одной или нескольких товарных строк и/или наличных на выбранное место и открывается только command action на home;
- экран `Курьеры` у коммерческого руководителя и внутренний режим `Остатки → Курьеры` у Директора показывают read-only товарные и наличные балансы курьеров;
- admin UI для courier sale/load/unload намеренно не добавлен, хотя backend поддерживает admin flow для write-команд с явным курьером.

Фактический директорский control/stock UI:

- главная Директора является эталоном для контрольных экранов: белые панели, тонкие ledger-разделители, компактная типографика, restrained radii, без decorative KPI/cards и без данных ради данных;
- `Остатки Директора` являются эталоном stock ledger: fixed topbar, embedded segmented tabs, компактная верхняя strip-сводка, white ledger panels and owner/detail rows;
- tab controls внутри директорских control surfaces встроены в панель: muted inactive surface, active white fill, тонкая рамка, без gap между active tab и divider;
- `Остатки → Распределитель` показывает материнскую строку распределителя, табличные колонки `Наименование / Количество / Итого`, full-width action `Списать наличные` и inline action `Снизить`;
- `Остатки → Курьеры` не повторяет labels под каждым курьером: есть одна шапка `Курьер / Остаток / Наличные`, далее owner row курьера и облегченные product detail rows;
- новые stock/read-only поверхности должны сначала пытаться повторить owner/detail ledger pattern, а не возвращаться к декоративным карточкам.

Фактический priced stock / discount UI:

- `DistributorStockList` показывает главным числом фактическую цену строки `unitPriceCents`;
- если строка дисконтирована, компактная иконка снижения показывается рядом с названием продукции, без видимого текстового badge, а базовая цена партии выводится вторичной зачеркнутой строкой;
- в sale product select для распределителя и курьера дисконтированные priced rows помечаются той же иконкой рядом с названием продукции;
- sale combobox для клиента и продукции открывается по focus и до ввода показывает первые 3 доступных варианта;
- клиентский combobox и product select используют общий search-combobox каркас: `listbox/option`, одинаковая очистка выбранного значения, одинаковые loading/empty states и фильтрация по введенному тексту;
- выбранный клиент в sale combobox отображается как `имя · телефон`, чтобы пользователь мог отличить однофамильцев до отправки операции;
- обзорные метрики, read-only stock lists, production read-only цены и operation summaries показывают деньги compact format без лишних `.00`; audit/detail views и денежные input values сохраняют exact two-decimal format;
- у Директора в режиме `Остатки → Распределитель` есть inline action `Снизить цену` рядом с товарной строкой;
- форма дисконта открывается от конкретной строки остатка и отправляет только `distributorProductBalanceId`, количество, новую цену и optional комментарий;
- submit дисконта disabled при offline, некорректном количестве, цене `<= 0`, цене не ниже текущей или количестве сверх остатка;
- после успешного дисконта dialog показывает inline result layer `Записано` с продукцией, компактной строкой `Снижено` в формате `количество • старая цена → новая цена`, общей скидкой и остатком по старой цене только если он больше нуля; `Готово` закрывает dialog, `Снизить еще` возвращает к списку без global success notice;
- работник распределителя, коммерческий руководитель и курьер видят priced stock как свойство товара, но не могут изменить цену в sale/load/unload flow;
- sale/load/unload screens считают totals по `unitPriceCents` выбранной строки и не отправляют цену продажи на backend.

- пункт `Экспорт` можно показывать как будущий системный entry point только в честном non-clickable состоянии, пока reports/export flow не реализован; смена собственного пароля уже доступна в account-блоке `Еще` для всех ролей;
- `Каталог`, `Клиенты`, `Профиль`, `Аналитика`, `Распределитель` и `Курьеры` не выводятся отдельными нижними пунктами у Директора;
- fake-блоки `Продажи сегодня`, `Последние операции`, fake revenue/report numbers не добавлять до backend/read model;
- управленческие действия `Назначить дисконт`, `Отчеты`, `История` не показываются как disabled tiles на home, пока backend flow не реализован;
- bottom navigation директора не показывает `Продажа`, потому что у роли нет write sale action.

Фактический admin users screen:

- `apps/web/src/roles/admin/AdminHome.tsx` является тонким wrapper и подключает `features/users/AdminUsersHome.tsx`;
- экран начинается обычным заголовком `Пользователи` с количеством сотрудников и компактной кнопкой `Новый`, без decorative summary-card;
- поиск по сотрудникам и фильтр роли находятся в компактном toolbar над списком; для v1 фильтрация client-side, потому что `/users` уже возвращает текущий список сотрудников целиком;
- форма создания сотрудника открывается по действию `Новый` в `operation-dialog` modal поверх списка и закрывается после успешного создания;
- временный пароль показывается отдельным compact notice с возможностью копирования и закрытия;
- сотрудники отображаются как плотный access-list в white management panel, а не как `entity-card`: строка показывает имя, login, role select, edit identity action, copy login action и reset action;
- edit identity открывает `operation-dialog` с текущим именем и login, сохраняет через `PATCH /users/:userId/identity`, после успеха обновляет `["users"]` и не меняет роль/пароль;
- copy login, подтвержденная смена роли и сохранение identity показывают global success notice над bottom navigation и не переводят пользователя на другой экран;
- смена роли не отправляет PATCH из select сразу: выбор открывает `operation-dialog` confirmation, потому что роль меняет доступы сотрудника;
- сброс пароля требует подтверждения в `operation-dialog` modal и после успеха обновляет `["users"]`;
- admin write-действия в users screen блокируются offline на первом уровне: создание, изменение имени/login, смена роли и сброс пароля визуально недоступны без сети;
- admin bottom navigation использует `Еще`, а не старую вкладку `Настройки`; account block показывает смену собственного пароля и logout row в том же ledger vocabulary, что остальные роли;
- роль admin не получает операционные sale/load UI, хотя backend может поддерживать часть этих flows.

Фактический operation history screen:

- `apps/web/src/features/operations/OperationHistoryHome.tsx` является read-only feature-компонентом для `admin` и `director`;
- нижняя навигация показывает вкладку `История` только при permission `operation.history.read`;
- admin и директор используют одно название `История`, без технического слова `Аудит` в UI;
- экран загружает список через `GET /operations/history`, а варианты фильтров через отдельный `GET /operations/history/options`;
- верхняя область с заголовком, компактной строкой периода и кнопкой `Фильтры` остается закрепленной внутри экрана, а скроллится только список операций; полный набор фильтров — период, событие `operationType`, пользователь, роль и источник — открывается в Radix Dialog поверх экрана, чтобы список событий оставался первым рабочим объектом; период внутри фильтров использует компактные date inputs текущей платформы;
- список использует плотные строки с разделителями, а не декоративные карточки: главное действие, actor/роль/источник/дата, сумма и количество;
- на узких мобильных экранах строки истории используют отдельную двухстрочную mobile-композицию: действие и переход остаются в верхней строке, сумма/количество — вторичной строкой, без падения иконки перехода в левый край;
- кнопка `Показать еще` использует backend cursor and append pagination: следующая страница добавляется ниже уже загруженных строк, а не заменяет текущий список;
- выбранные фильтры истории операций сохраняются в browser local storage, чтобы пользователь мог уйти со вкладки и вернуться к тому же расследованию; быстрые preset-периоды не используются;
- details открываются через Radix Dialog в общем `operation-dialog` стиле и отображаются через `operation-detail-presenter.ts` как управленческие секции: участники, товар, деньги, остатки, комментарий/причина и изменения;
- модалка не показывает technical ids, balance row ids, raw internal keys, normalized fields и значения `[redacted]`; audit storage при этом остается полным на backend;
- неизвестные будущие details проходят через fallback presenter: id-like/sensitive keys скрываются case-insensitive, nested object/array показываются как повторяющиеся человекочитаемые группы, а не raw JSON;
- frontend не выполняет security redaction самостоятельно как источник истины: backend уже маскирует details, UI дополнительно фильтрует технический шум для директорского экрана.

## 10. Component Architecture

Frontend структура должна помогать быстро находить код и не превращать role screens в большие файлы.

Принципы:

- route-level `page.tsx` должен быть тонким wrapper, а не местом бизнес-логики;
- role home component должен собирать экран из секций и feature-компонентов;
- бизнес-функции, которые переиспользуются ролями, живут в `features/*`, а не внутри конкретной роли;
- роль подключает feature в нужном контексте, но не копирует форму или логику;
- общие визуальные primitives живут в `ui/*`;
- `shared` package не использовать для React-компонентов.

Пример переиспользования:

- продажа нужна курьеру, работнику распределителя и коммерческому руководителю;
- продажа распределителя живет в `features/sales/DistributorSaleHome.tsx` и открывается из `CommercialManagerHome` / `DistributorWorkerHome` через общий distributor action;
- курьерская продажа живет в `features/courier/CourierSaleHome.tsx`, потому что работает с courier balance и своим API;
- история продаж общая для распределителя и курьера: `features/sales/SalesHistoryHome.tsx` + `RecentSalesPanel.tsx`;
- backend финально проверяет права и источник продажи, frontend только подстраивает форму под контекст роли.

Граница ответственности:

- `roles/*` — стартовые экраны ролей и композиция секций;
- `features/*` — бизнес-сценарии и переиспользуемые feature-компоненты;
- `app-shell/*` — оболочка приложения, навигация, online/offline state;
- `auth/*` — login/logout/current actor frontend flow;
- `ui/*` — визуальные primitives без доменных правил.

Фактический control rhythm и reusable primitives:

- `DESIGN.md` является главным визуальным SoR; `DirectorAnalyticsHome` сейчас является практическим эталоном плотной контрольной поверхности для dashboard/read-only/report screens;
- новые read-only/control экраны должны начинаться с языка директорской главной: верхняя смысловая сводка, затем табличные панели с тонкими разделителями и компактной типографикой, без повторяющихся карточек и декоративных статусов;
- CSS-селекторы должны быть изолированы по экрану или feature prefix, чтобы точечная полировка одной роли не меняла соседние экраны; при этом локальный слой обязан использовать общие стандарты из `DESIGN.md`: шкалу шрифтов, treatment вкладок, button vocabulary, spacing rhythm, line colors и state colors;
- если экрану нужен новый локальный selector scope, он должен описывать композицию конкретного экрана, а не изобретать новые размеры, radii, тени и цвета; повторившийся паттерн после 2-3 экранов нужно выносить в shared primitive или документированный token;
- `apps/web/src/ui/SegmentedControl.tsx`: общий primitive для внутренних вкладок и segmented controls. Он сам выставляет количество колонок через `--segmented-control-columns`, поддерживает `role="tablist"` и `role="group"`, optional lucide icon и `aria-selected` / `aria-pressed`;
- segmented container: `border: 1px solid var(--line)`, `background: var(--base-white)`, `border-radius: 12px`, `padding: 4px`, `gap: 6px`;
- segmented option: `min-height: 36px`, `border-radius: 9px`, `font-size: 12px`, `font-weight: 400`, `gap: 5px`, `padding-inline: 8px`; active state: black fill + white text, hover: `#f5f7f4`, focus ring: `2px solid var(--focus-ring)`;
- icon внутри segmented option: lucide, default `14px`; на узких catalog tabs до `430px` option становится `42px` высотой, text `11px`, icon остается `14px`;
- базовые form fields используют `--field-height: 46px`, `--control-radius: 8px`, `--field-bg: #fbfcfb`, `border: 1px solid var(--line)`;
- `management-surface`: reusable scope для плотных CRUD-экранов со списками, формами, actions и confirmations: primary/secondary/status buttons `42px` высотой, `8px` radius, `13px` text, `12px` horizontal padding; compact buttons `40px`; icon buttons `42x42px`; mobile touch target в формах `44px`;
- form panels остаются `border: 1px solid var(--line)`, `border-radius: 8px`, `padding: 14px`; на плотной mobile форме можно снижать padding до `12px`, если сохраняется 44px touch target;
- destructive active action (`В архив`) использует `--state-danger-border/bg/text`, restore/success action использует `--state-success-border/text` + `var(--brand-lime)`, чтобы active destructive не выглядел disabled;
- новые экраны, где нужен похожий CRUD/list/form pattern, должны сначала переиспользовать `SegmentedControl`, `.form-panel`, `.entity-actions`, `.management-surface`, `.primary-button`, `.secondary-button`, `.secondary-icon-button`, `.status-button`, а не заводить локальные radius/height для каждой формы;
- новые экраны, где нужен dashboard/control/list pattern, должны сначала сверяться с `director-dashboard-*` extraction и `DESIGN.md` Dashboard / Control Surfaces, а не копировать старые декоративные карточки или заводить новые screen-local токены;
- стандарты action blocks, новых компактных кнопок и role-specific command areas еще не завершены: до extraction такие блоки можно держать локально, но они должны быть визуально совместимы с текущими tabs/buttons/spacing и не должны возвращать старый декоративный стиль.

Текущий catalog foundation реализован как общий feature в `apps/web/src/features/catalog/CatalogHome.tsx` и подключен через mobile shell для пользователей с `catalog.manage`, `catalog.raw_material.manage` или `catalog.packaging.manage`. Экран справочников не принадлежит только роли администратора: администратор и директор используют один feature со всеми вкладками, а заведующий производством открывает тот же feature из `Еще → Справочники`, но видит только вкладки `Сырье` и `Тара`. Шаблоны продукции, цены и распределители остаются в UI только у ролей с `catalog.manage`, то есть у директора и администратора. Экран имеет один heading `Справочники`; текущий контекст задают текстовые segmented tabs `Сырье`, `Тара`, `Распределители`, `Шаблоны` в том же визуальном rhythm, что внутренние директорские табы, без повторного section heading внутри таба. Каждый tab сначала показывает компактную строку count + действия `Добавить` / `Архив`, затем список. Catalog route доступен, если actor имеет `catalog.manage` или хотя бы одно granular catalog permission; набор tabs строится по permissions текущего actor. Каталог подключает reusable `SegmentedControl` и `management-surface`, поэтому кнопки, icon-buttons, radius, высоты и action grid совпадают с control rhythm выше. На ширинах 320-430px строка справочника остается одной горизонталью: название и детали слева, компактная группа действий справа; активное `В архив` в строке становится icon-only, архивные строки показывают текстовое `Вернуть`, без disabled edit action.

Создание и редактирование справочных записей открываются в Radix Dialog поверх экрана, а не раскрываются inline внутри списка. Это сохраняет плотный ledger rhythm списка и одинаковую модель действий для простых сущностей и `Шаблоны`. Архивирование активной строки также требует modal confirmation: строка остается стабильной, заголовком dialog становится имя архивируемой записи, а destructive action подтверждается кнопкой `Архив`. После успешного архива экран показывает status-строку с быстрым `Вернуть`, чтобы восстановление не требовало перехода в архив и не сбрасывало текущий tab. Write-действия на экране справочников отключаются при offline state.

Форма шаблона продукции в tab `Шаблоны` визуально группирует поля внутри dialog: `Название шаблона`, затем `Сырье` и `Тара`, затем `Цена за единицу`. Этот же layout используется в create и edit state, без вложенных карточек.

Текущий production baseline реализован как feature в `apps/web/src/features/production/ProductionHome.tsx` и подключен через `RoleHomeRouter` для роли `production_manager`.

Фактическая нижняя навигация заведующего производством:

- `Главная` — белая панель `Цех` со строками `Продукция`, `Сырье` и `Тара`, затем 4 действия без встроенных списков;
- `Распределитель` — read-only экран `На распределителе`;
- `Уведомления` — рабочий экран `Задачи производству` со списком задач, polling 30 секунд и checkbox completion для новых задач;
- `Еще` — меню с `Справочники`, `История`, account-блоком, сменой пароля и выходом.

Фактический notification flow:

- `NotificationsHome` используется для production/commercial экранов и остается feature-компонентом;
- commercial manager открывает экран задач через home action `Уведомить`, видит общий список задач производству и форму `Новая задача`;
- форма `Новая задача` использует общий operational form standard: видимый label `Что передать производству`, счетчик 0/1000, placeholder с примером и submit `Отправить задачу`;
- production manager открывает тот же список через вкладку `Уведомления`, но без формы создания;
- production manager видит компактный счетчик новых задач на пункте нижней навигации `Уведомления`;
- по умолчанию открыт список `Новые`; выполненные задачи скрыты из основной очереди и доступны через segmented switch `Новые / Выполненные`;
- production manager отмечает новую задачу checkbox control с aria-label `Отметить задачу выполненной`;
- список обновляется через React Query polling каждые 30 секунд, refetch on focus и invalidation после create/complete;
- write-действия disabled offline и показывают короткую причину блокировки.

Production account flow находится в `Еще`: экран использует тот же Director More menu vocabulary, role/account данные показываются только в account-блоке. Смена собственного пароля доступна из общего account section для всех ролей, открывается в `operation-dialog`, требует текущий пароль, новый пароль и повтор, а после успеха показывает inline success без завершения текущей сессии.

Нижняя навигация визуально icon-only, но каждый пункт сохраняет `aria-label` с названием раздела.

Визуальный ориентир production home:

- основной шрифт — Helvetica;
- веса шрифта задаются через CSS-токены: `400` regular/labels, `500` emphasis, `600` только как резерв для редких сильных акцентов;
- панель `Цех` белая, с тонкими ledger-разделителями, compact typography и restrained radius;
- на главной нет крупного hero metric, lime hero-card и декоративных action tiles;
- быстрые действия одинаковые по форме: `Выпустить` и `Передать` отделены от `Добавить сырье` и `Добавить тару` пространством и тонким разделителем, без черной заливки как локального акцента;
- черный цвет остается для нижней навигации, primary submit-кнопок и selected states, а не для плиток быстрых действий;
- иконки в сводке сидят в мягких зеленых квадратах с согласованным скруглением, без острых внутренних углов;
- production iconography не дублирует смысловые уровни: готовая продукция — `BadgeCheck`, сырье — лаконичный `FishSymbol`, тара — `Package`, выпуск — `Factory`, добавление тары — `PackagePlus`, добавление сырья — нейтральный `CirclePlus`.

Кликабельные зоны карточки `Цех` открывают отдельные экраны с кнопкой `Назад`:

- `Продукция` — список готовых партий в цеху;
- `Сырье` — виды сырья с ненулевым остатком и количество по каждому виду;
- `Тара` — виды тары с ненулевым остатком и количество по каждому виду.

Быстрые действия на главной производства открывают отдельные короткие экраны с кнопкой `Назад`:

- `Выпустить` — только форма выпуска продукции;
- `Передать` — форма `На распределитель`, перемещение готовой продукции из цеха на активный распределитель;
- `Добавить сырье` — только форма прихода сырья;
- `Добавить тару` — только форма прихода тары.

Цена шаблона продукции вводится в справочниках в рублях, но в API отправляется как `priceCents`. Production release использует активные шаблоны с ценой и показывает цену как snapshot будущей партии.

Форма `Выпуск продукции` после выбора шаблона показывает компактную строку доступности: сколько сырья и тары сейчас есть в цеху для выбранного шаблона. Если выбранному шаблону явно не хватает сырья или тары, форма показывает warning до submit и блокирует кнопку выпуска. Остаток после выпуска в этой форме не показывается, чтобы не перегружать рабочий экран.

Экран “Продукция в цеху” показывает доступный остаток готовой продукции из `GET /production/workshop-product-balances`, а не все выпущенные партии. История по-прежнему показывает последние выпуски через `GET /production/product-batches`; read-only выпуски отображаются как плотный table/list без карточек, потому что строка истории не открывает detail-screen.

Форма `Передать` использует `GET /production/transfer-options`, показывает доступное количество, цену партии и дату выпуска. Submit отправляет `POST /production/product-transfers`, блокируется offline и во время pending-состояния, а после успеха обновляет production summary, остатки цеха, transfer options и историю.

Production write actions используют проверяемый результат по уровню риска. Приход сырья и тары остаются короткими операциями: после успешного ответа backend экран возвращается на главную производства и показывает compact success notice снизу над навигацией. Выпуск продукции и передача на распределитель показывают inline result layer `Записано` с продукцией, количеством, расходом/ценой/получателем и actions `Готово` / `Новый выпуск` или `Готово` / `Новая передача`; старый bottom notice для этих операций не показывается. Spinner-first feedback не используется для быстрых операций; pending-состояние кнопок остается только защитой от двойного submit. Ошибки остаются inline в форме и не возвращают пользователя на главный экран. На главной производства write action tiles `Выпустить`, `Передать`, `Добавить сырье` и `Добавить тару` блокируются offline уже до входа в форму.

Post-submit result layer является общим frontend pattern для операций с деньгами и остатками. Короткие справочные действия могут оставаться на compact success notice, но продажи, загрузки, возвраты, выпуск, передача продукции, списания, дисконты и отмены продаж не должны завершаться только transient toast.

Success notice не должен быть черным, потому что появляется над черной нижней навигацией. Базовый визуальный вариант — светлый `brand.lime` фон с темным текстом и короткой иконкой успеха, без кнопки закрытия и без action.

Production forms по-прежнему используют read-only endpoints под `production.manage` для рабочих операций выпуска, прихода и перемещения. Отдельно заведующий производством может открыть `Еще → Справочники`, где `CatalogHome` читает и меняет только `/catalog/raw-material-types*` и `/catalog/packaging-types*` по granular permissions `catalog.raw_material.manage` и `catalog.packaging.manage`; `/catalog/distributors*` и `/catalog/product-templates*` в этом UI для него не запрашиваются и не показываются.

Для `production_manager` во вкладке `Еще` есть пункты `Справочники` и `Как работает цех`. `Справочники` открывают существующий `CatalogHome` с вкладками `Сырье` и `Тара`; в этих вкладках доступны просмотр, добавление, редактирование, архивация, просмотр архива и восстановление. `Как работает цех` открывает frontend-only screen `RoleOnboardingHome`: шаги перелистываются назад/вперед, возврат идет через нижнюю навигацию, а отдельный крестик закрытия не показывается. Нижняя навигация сохраняет active state `Еще`. Экран не использует вложенную demo-карточку: каждый шаг состоит из фрагмента реального production UI с примерными данными (`Икра А`, сырье, тара, распределитель, result layer), выделенного зеленой рамкой без заливки, отдельной текстовой карточки без заголовков/жирных акцентов и controls перелистывания. Контент шага скроллится в отдельной области, а controls перелистывания остаются отдельным footer-блоком над нижней навигацией и не скроллятся вместе с текстом. Для операций допускаются соседние фрагменты точки входа и заполненной формы, например отдельная кнопка `Добавить сырье`/`Добавить тару`/`Выпустить`/`Передать` и ниже форма операции без повторяющего заголовка. Он не вызывает write API, не пишет audit/history и не меняет рабочие балансы. Последний шаг про ограничения роли является чисто информационным: без отдельного visual-фрагмента, формы и submit-кнопки, он сообщает, что заведующий производством сам управляет видами сырья и тары, но шаблоны продукции, цены и распределители добавляют и меняют только директор или администратор.

Для `commercial_manager` во вкладке `Еще` есть пункт `Как работают продажи`. Он использует тот же `RoleOnboardingHome` и тот же visual pattern: фрагмент настоящего интерфейса с примерными данными, зеленая рамка без заливки, отдельная текстовая карточка и фиксированные controls над нижней навигацией. Шаги показывают сводку распределителя, продажу с клиентом/товаром/оплатой, раздел клиентов, read-only контроль балансов курьеров и задачу производству. В текстах явно закреплено, что коммерческий руководитель контролирует курьерские балансы, но не оформляет загрузку или возврат за курьера. Онбординг коммерческого руководителя также полностью frontend-only: не вызывает write API, не пишет audit/history и не меняет рабочие остатки.

Read-only экран остатков распределителя живет в `features/distributor/DistributorInventoryHome.tsx` и читает `GET /distributor/inventory`. Для `production_manager` он использует variant `stock-ledger`: над списком белая сводка `Количество` / `Продукция`, без наличных, скидок и cash actions; количество позиций показывается в meta шапки таблицы, но не дублируется в верхнем heading. Позиции ниже отображаются как плотный read-only table/list с колонками `Наименование`, `Количество`, `Итого`, без карточек и иконок, чтобы не создавать ложное ожидание перехода внутрь позиции. На ширинах 320-430px строки сохраняют видимые заголовки колонок, а цена не проваливается под название продукции. Для ролей с `distributor.cash.read` этот экран дополнительно читает `GET /distributor/cash-balances` и показывает наличный баланс распределителя. Для ролей без этого permission cash balance не запрашивается и не отображается.

Для Директора в режиме `Остатки → Распределитель` этот же feature показывает компактное действие `Списать наличные`. Действие не раскрывает форму и показывает короткую причину, пока наличные загружаются, нет активного распределителя или нет наличных для списания. Форма выбирает только активные распределители из `GET /distributor/cash-balances`; если активный распределитель один, он выбран автоматически. Комментарий optional. Operation summary показывает доступную сумму, сумму списания и остаток после списания. Submit подписан `Подтвердить списание`, чтобы не дублировать кнопку открытия формы. После успеха dialog остается в контексте операции и показывает result layer `Записано` с распределителем, суммой списания и остатком наличных; `Готово` закрывает dialog, `Списать еще` возвращает форму списания без global success notice. Offline, нулевая сумма, сумма сверх cash balance и отсутствие активного распределителя блокируют submit.

Для `production_manager` экран подключен отдельной нижней вкладкой `Распределитель` и открывается с заголовком `На распределителе`, чтобы не смешивать production home с read-only остатками распределителя. Для остальных не-admin ролей с `distributor.stock.read` home показывает distributor inventory с заголовком `Остатки`. `admin` имеет backend read access, но отдельный admin UI для этого этапа не добавлен.

Экран клиентов живет в `features/clients/ClientsHome.tsx` и подключается для ролей с `client.read`. У Директора вход находится в `Еще → Клиенты`, чтобы не расширять нижнюю навигацию; у ролей, где клиенты являются частым рабочим разделом, он может оставаться отдельной вкладкой. Экран начинается обычной heading row: `Клиенты` слева, количество клиентов и компактное действие `Добавить клиента` справа, без summary-карточки. Основной поток здесь поиск и список, поэтому создание и редактирование не должны разрывать search-results зону: роли с `client.manage` открывают формы клиента в `operation-dialog` поверх списка, по тому же modal pattern, что директорские справочники. Поиск работает live/debounce по имени или телефону, без ручной кнопки `Найти`; в поле есть очистка активного поиска. Клиенты отображаются как плотный список с верхней/нижней ledger-линией: имя, телефон и muted-описание слева, быстрый copy phone справа, а роли с `client.manage` дополнительно видят редактирование. Директор видит read-only список и поиск. `production_manager` вкладку клиентов не видит. После успешного создания или редактирования dialog закрывается, пользователь остается в списке и видит compact success notice: `Клиент добавлен` или `Клиент обновлен`. Backend errors остаются inline в dialog форме.

Экран продажи с распределителя живет в `features/sales/DistributorSaleHome.tsx` и открывается command action `Продать` для ролей с `distributor.sale.create`, кроме `admin`, для которого UI продажи пока не добавлен. Для коммерческого руководителя форма открывается как detail screen с `Назад` на главную `Продажи`; для работника распределителя используется тот же shared sale flow. Форма следует общему operational form standard: белые `form-panel`, тонкие ledger-разделители, compact heading, отдельная панель `Клиент` и панель `Детали продажи`. Клиент выбирается через searchable combobox на базе `GET /clients?search=...`; при пустом поле combobox показывает первые 3 клиента. Кнопка `Новый клиент` открывает modal dialog поверх формы продажи по директорскому `operation-dialog` pattern с затемнением фона; после `POST /clients` созданный клиент выбирается в продаже, а dialog закрывается. После выбора клиента кнопка `Новый клиент` скрывается до очистки выбранного клиента. Затем пользователь выбирает товарную строку из `GET /distributor/sale-options`; product select при пустом поле показывает первые 3 доступные товарные строки. Продажа отправляется в `POST /distributor/sales` по `distributorProductBalanceId`, количеству и способу оплаты. Способ оплаты выбирается через общий `PaymentMethodSegmentedControl`, как в курьерской продаже. Верхняя decorative summary-card, склеенная строка доступности и список последних продаж не используются: доступность/цена выбранной позиции показываются ledger-строками, общий расчет находится в нижней form ledger перед submit, а работа с уже созданными продажами вынесена из формы в `Историю продаж`. После успеха экран остается в продаже и показывает inline result layer `Записано` с клиентом, компактной строкой `Продано` в формате `продукция • количество • цена`, оплатой и итогом; `Готово` возвращает на home без global success notice, `Новая продажа` открывает новую форму. Ошибки остаются inline, offline блокирует submit; если submit disabled, рядом с кнопкой показывается короткая причина блокировки.

Экран `История продаж` живет в `features/sales/SalesHistoryHome.tsx`. У коммерческого руководителя и курьера вход находится в `Еще → История`, чтобы нижняя навигация оставалась на частых рабочих разделах; у работника распределителя история остается отдельной нижней вкладкой. На экране не дублируется внутренний заголовок `Последние продажи`: один экранный заголовок задает контекст. Экран читает paginated endpoints `GET /distributor/sales/history` или `GET /courier/sales/history` с `limit`, `cursor`, `search` и `status`. Поиск один: клиент, телефон или товар; статус переключается `Все / Активные / Отмененные`. Быстрые периоды и export/print в этот flow не входят. Строка показывает продукцию, клиента, количество, фактическую цену, сумму, способ оплаты, автора продажи и статус отмены; список оформлен как compact ledger с верхней/нижней линией и без декоративных карточек. Сумма и действие вынесены в отдельную правую колонку, чтобы не смешиваться с названием и деталями. `Показать еще` добавляет следующие продажи ниже текущих строк. Неотмененная продажа имеет вторичное действие `Отменить`; оно открывает `operation-dialog` с затемнением фона, краткой сводкой продажи и полем `Причина отмены`, затем отправляет `POST /distributor/sales/:saleId/cancel` или courier cancel endpoint. Причина обязательна, offline и pending блокируют submit. После успешной отмены dialog закрывается, пользователь остается в `Истории продаж`, видит inline result layer `Продажа отменена` с клиентом, компактной строкой `Отменено`, суммой со способом оплаты и причиной; обновляются inventory/sale-options/cash/sales history queries, отмененная строка становится приглушенной и больше не показывает action.

Экран курьерского баланса живет в `features/courier/CourierBalanceHome.tsx` и читает `GET /courier/product-balances`. У курьера home показывает только собственный товарный баланс. У коммерческого руководителя вкладка `Курьеры`, а у Директора режим `Остатки → Курьеры`, показывает read-only балансы всех курьеров: сверху один компактный белый блок `Количество`, `Продукция`, `Наличные` с тем же плотным one-line metric pattern и compact money format, ниже без повторного заголовка идет отдельный `courier-balance-list`. Commercial и Director используют один owner/detail courier ledger variant: глобальная шапка `Курьер / Остаток / Наличные`, затем материнская строка курьера с именем, количеством позиций, стоимостью продукции и наличными, далее облегченные товарные detail rows. Login handle, отдельный общий список `Товары`, большую summary-карточку, отдельную карточку наличных, per-courier footer `Всего продукции` и два параллельных списка по курьерам на этом экране не использовать. На ширинах 320-430px остаются видимыми заголовки колонок, а товарные строки не проваливают цену под название продукции. `admin` имеет backend read access, но отдельный admin UI для этого этапа не добавлен.

Экран курьерской продажи живет в `features/courier/CourierSaleHome.tsx` и использует тот же sale form pattern, что распределитель: клиентский combobox с первыми 3 клиентами при пустом поле, `Новый клиент` в `operation-dialog` поверх формы, product select с первыми 3 собственными товарными строками, ledger-строки `Доступно` / `Цена` для выбранной позиции, segmented control для cash/cashless, плоский form ledger итогов и короткую причину блокировки рядом с disabled submit. Список последних продаж не показывается внутри формы продажи. У курьера `История` открывается через `Еще`, использует общий `SalesHistoryHome`, читает `GET /courier/sales/history` и показывает только продажи текущего курьера. Отмена собственной продажи требует причину и отправляет `POST /courier/sales/:saleId/cancel`; после успеха обновляются courier product balances, sale-options, cash-balances и sales history. Коммерческий руководитель и Директор не видят cancel action для курьерских продаж.

Экран загрузки курьера живет в `features/courier/CourierLoadHome.tsx` и открывается command action `Загрузить` только для роли `courier`. Форма выбирает строку товара из `GET /courier/load-options`; если строка дисконтирована, product select показывает тот же маркер сниженной цены, что и sale flows. Выбранная позиция отображается ledger-строками `Доступно` / `Цена`, итог операции показывается form ledger строками `Количество`, `Итого`, `Операция`. Submit отправляет `POST /courier/loads` без `courierUserId`. Верхняя decorative summary-card не используется. После успеха экран показывает inline result layer `Записано` с продукцией, распределителем, строкой `Загружено` в формате `количество • цена`, итогом и новым остатком курьера только когда он не повторяет загруженное количество; `Готово` возвращает на home без global success notice, `Новая загрузка` открывает новую форму. Ошибки остаются inline, offline и pending-состояние блокируют submit; если submit disabled, рядом с кнопкой показывается короткая причина блокировки.

## 11. Что остается вне main-role cleanup

- cross-role hardening: смена пароля, reports/control pack, desktop/tablet expansion;
- детальная component API для будущих бизнес-функций до появления самих функций;
- новые роли или новые read models, которых еще нет в runtime/API.

## 12. Зафиксированные frontend conventions

- layout conventions: mobile shell, bottom nav, screen headings, white ledger panels;
- form patterns: operational `production-action-form`, `operation-dialog`, shared combobox/select controls and inline block reasons;
- table/list patterns: ledger rows, visible column headers, no decorative cards for read-only data;
- error/loading/empty states: inline errors, compact loading text, no fake metrics for empty data;
- component test strategy: targeted role-flow tests for navigation, forms, dialogs and query invalidation.
